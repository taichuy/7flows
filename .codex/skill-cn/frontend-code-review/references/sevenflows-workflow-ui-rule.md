# 7Flows 前端工作流 UI 审查规则

在审查 `web/` 目录时，将本规则与通用前端规则一起使用。重点不是复刻 Dify 的界面，而是确保实现符合 7Flows 的产品定位与未来编辑器架构。

## 必读上下文

以下主题被修改时，先阅读文档：

- 画布、节点、发布、调试、用户心智：`docs/product-design.md`
- 插件 UI、变量传递、节点调试、值缓存、安全：`docs/technical-design-supplement.md`
- 当前前后端基线和运行时边界：`docs/dev/runtime-foundation.md`

## 当前仓库现实

- 当前前端还是 Next.js App Router 的轻量骨架：`web/app/`、`web/components/`、`web/lib/`
- 还没有正式落地 xyflow 画布、节点配置抽屉、调试面板、发布配置页
- 因此审查时既要约束“未来该怎么做”，也要警惕假装功能已完整可用的 UI

## 7Flows 特有关键规则

### 1. 这是多 Agent 工作流平台，不是 Prompt 表单壳

- 节点配置不能只剩 prompt、模型和提交按钮。
- 对于 LLM Agent / Tool / Sandbox / MCP / Router / Loop / Output 等节点，界面应逐步显式表达：
  - 模型供应商
  - 工具开关
  - MCP 能力
  - 沙盒策略
  - 上下文授权
  - 输入输出 schema
  - 超时 / 重试 / 运行策略

### 2. 调试与发布是一级能力，不应永远藏在次级弹窗里

- 产品文档明确要求运行调试面板、发布配置页、凭证管理、插件管理是首版核心界面。
- 审查时若发现设计把这些能力弱化为“以后再说”的补充弹窗，需要指出风险。

### 3. 内部以 `7Flows IR` 组织心智，外部协议只做映射

- UI 文案、状态和数据结构应优先围绕 workflow / node / edge / run / node run / published endpoint。
- 不要把 OpenAI / Anthropic / Dify 插件字段直接变成页面主心智。
- 发布协议界面应体现“映射到外部协议”，而不是“平台内部就是 OpenAI”。

### 4. 节点权限与安全必须有可见性

- 上下文授权不是隐藏实现细节，应在节点配置中有明确可见的来源与范围。
- 凭证、沙盒、危险工具、文件访问等高风险能力，需要有显式开关、状态提示或权限说明。
- 如果 UI 默认让节点读取全部前序数据、或对敏感能力没有任何可见提示，应判为高风险设计问题。

### 5. xyflow 集成要遵守稳定引用和交互边界

- `nodeTypes` / `edgeTypes` 需要稳定引用，避免在渲染期间重建。
- 节点内部表单控件应加 `nodrag` / `nowheel` 一类交互保护。
- 节点壳层、配置表单、动态 schema 渲染器、调试视图应拆层，而不是一个节点组件包办所有逻辑。

### 6. 不要把 Dify 的 store / 组件假设直接搬过来

- 当前仓库没有 Dify 的 `workflowStore`、`service/use-*.ts`、`analyze-component` 等基础设施。
- 审查时如果改动依赖这些并不存在的模式，需要指出它与当前代码库不兼容。
- 在 7Flows 当前阶段，应优先遵循 Next.js App Router、局部组件拆分和真实存在的目录结构。

### 7. MVP 边界要诚实表达

- 当前 runtime 还不支持完整 loop、正式 MCP、插件代理和发布协议映射。
- 前端不应把未完成能力渲染成“已经可用”的假象；应有 disabled、coming soon、实验态或占位说明。
- 视觉上可以前瞻，但交互上不能误导。

## 审查优先级建议

1. 是否把 7Flows 做成了“Dify 风格聊天壳”而不是工作流平台
2. 是否遗漏了节点权限、沙盒、调试、发布这些核心能力的可见性
3. 是否违反 xyflow 集成边界，导致拖拽/重渲染/状态管理风险
4. 是否把未落地功能伪装成已可用
5. 是否引入当前仓库不存在的 Dify 约定或基础设施
