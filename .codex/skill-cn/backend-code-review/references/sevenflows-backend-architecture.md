# 7Flows 后端架构审查规则

在审查 `api/` 目录下的代码时，将本规则与通用后端规则一起使用。本规则优先关注 7Flows 的产品边界、运行时语义和当前仓库的演进阶段。

## 必读上下文

以下任一主题被修改时，先阅读对应文档再审查：

- 工作流定义、节点语义、发布协议：`docs/product-design.md`
- 插件代理、变量传递、调试、缓存、安全：`docs/technical-design-supplement.md`
- 当前运行时 MVP 边界：`docs/dev/runtime-foundation.md`

## 当前仓库结构

- `api/app/api/routes/`：HTTP 路由，只做输入解析、调用服务、返回响应
- `api/app/schemas/`：Pydantic 传输模型
- `api/app/services/`：当前主要的工作流编排与运行逻辑
- `api/app/models/`：数据库实体
- `api/app/repositories/`：持久化抽象，复杂查询和数据写入应逐步收敛到这里
- `api/app/core/`：配置、数据库、Celery 等基础设施

## 7Flows 特有关键规则

### 1. 以 `7Flows IR` 为中心，而不是让适配层反客为主

- 新增的工作流、节点、运行时、发布相关逻辑，必须围绕 `Workflow`、`Node`、`Edge`、`RuntimeContext`、`AuthorizedContextRefs`、`PublishedEndpoint` 等概念收敛。
- 不要把 OpenAI / Anthropic / Dify 插件的协议对象直接扩散成内部主模型。
- 审查时重点识别：是否引入了第二套内部 DSL、第二套执行状态机、或协议专属字段污染核心模型。

### 2. 保持当前 MVP 边界，不要过早把所有能力揉进 `runtime.py`

- 当前 `api/app/services/runtime.py` 只是最小执行闭环，职责应集中在 DAG 排序、运行记录、事件产出与占位执行。
- 插件兼容、发布网关、MCP、沙盒、缓存、鉴权等新能力，不应直接塞进一个巨型运行函数；应拆到独立服务/执行器/适配器。
- 如果改动导致 `RuntimeService` 同时承担协议适配、权限判定、网络调用、沙盒管理、事件映射，视为高风险设计退化。

### 3. 循环必须显式表达，不能靠图结构“偷偷回边”

- 产品设计明确要求 `Loop` 节点显式表达循环。
- 任意隐式回边、绕过 `loop` 节点的循环、或在运行期偷偷重复调度节点，都会削弱可调试性和可观测性。
- 审查图执行逻辑时，要检查是否仍然坚持 “普通 DAG + 显式 loop” 的边界。

### 4. 上下文访问必须显式授权

- 节点默认不应读取全部前序节点输出。
- 任何跨节点读取都应经过 `AuthorizedContextRefs`、边映射、或显式 MCP 查询，而不是简单把 `outputs` 整包塞给后续节点。
- 如果代码把完整运行态上下文暴露给任意节点、插件或外部接口，应视为关键安全问题。

### 5. 发布协议是适配层，不是第二执行引擎

- `native`、`openai`、`anthropic` 只是发布层协议。
- 它们应映射到同一套内部运行逻辑和统一事件总线，而不是各自维护一套节点调度、状态转换和流式输出逻辑。
- 审查发布接口时，要重点检查是否出现复制 runtime 流程的分叉实现。

### 6. `run_events` 是调试和流式输出复用基座

- 运行时状态变化应通过统一事件流落库或统一分发。
- 不要在没有必要的情况下旁路 `run_events` 再造一套“仅前端可见”的调试数据格式。
- 新增流式能力时，应优先从统一事件模型映射，而不是在每个协议控制器里手写拼装。

### 7. 插件代理、沙盒、凭证必须保持清晰隔离

- Dify 兼容范围是“插件适配层”，不是完整 Dify ChatFlow DSL 兼容。
- `sandbox_code` 节点与高风险工具执行必须保留隔离边界，不能图省事直接宿主机执行。
- 凭证引用应保持“引用 + 运行时解密注入”的模式，不能把明文回写到工作流定义、节点配置或日志中。

### 8. 数据模型要区分设计态、运行态、发布态

- 工作流定义和运行记录不能混成一张“大表 JSON”。
- 当前仓库已经有 `workflows`、`runs`、`node_runs`、`run_events`；扩展时应延续这种按职责拆分的方向。
- 涉及运行态变更时，优先要求 Alembic 迁移，而不是重新引入 `create_all` 或启动时隐式建表。

## 审查优先级建议

1. 是否偏离 `7Flows IR`、显式授权、显式 loop、统一事件流这些核心原则
2. 是否让 MVP runtime 与未来插件/发布/沙盒职责耦合失控
3. 是否破坏当前运行态表结构、事件模型或迁移规范
4. 是否引入凭证泄漏、上下文越权或执行隔离缺失
5. 是否缺少对 `runs` / `node_runs` / `run_events` 行为的测试覆盖
