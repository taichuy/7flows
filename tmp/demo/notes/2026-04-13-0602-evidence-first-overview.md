# 1Flowse Demo 本轮诊断与计划

时间：2026-04-13 06:02
范围：仅修改 `tmp/demo`
状态：已完成

## 本轮批判

- `overview` 虽然已经把二级路径收回 `当前焦点`，但 hero 左侧仍主要是标题、说明和一个 CTA，桌面端形成大面积“无证据空场”。
- `当前焦点` 卡片承担了全部问题域说明，移动端进入首屏后会连续看到状态 badge、说明文案和三条动作，信息密度开始偏挤。
- 这不是简单的样式问题，而是 L0 概览层缺少一个“直接证明当前 flow 处于什么状态”的证据区，导致主内容只能靠说明文字补足。

## 信息架构判断

### 当前状态

| 区块 | L0 | L1 | L2 | 问题 |
| --- | --- | --- | --- | --- |
| hero 左侧 | 标题、说明、CTA | 无 | 进入编排 | 信息解释多，事实证据少 |
| 当前焦点 | 三条问题域摘要 | 跳转到 API / Logs / Monitoring | 对应任务页 | 单块承担过多说明责任 |
| 最近运行摘要 | run 列表摘要 | Drawer | Logs 页 | 结构正确，但被放在首屏下方，没参与 hero 叙事 |

### 修正目标

1. 在 hero 左侧补一个更具体的 `workspace pulse` 证据区，让首屏先回答“现在 flow 卡在哪里、发布面是否稳定、host gap 是否仍存在”。
2. `当前焦点` 保留为右侧问题域入口，但文案要更短，只负责“为什么值得去看”和“去哪处理”。
3. 新证据区应直接复用现有 mock 数据，避免再造一套独立叙事。

## 本轮方案

1. 新增 `workspace pulse` 区块，展示：
   - 当前 flow 的阶段序列
   - active run / published contract / embedded runtime 三个证据卡
2. 用测试锁定首屏必须同时具备：
   - `进入编排` 作为唯一主 CTA
   - `当前焦点`
   - `workspace pulse`
3. 根据新结构收紧 hero 文案和样式，降低移动端首屏高度。

## 预期结果

- 桌面端 hero 不再依赖大片留白撑布局，而是由“标题 + 证据 + 焦点”共同构成。
- 移动端在看到 CTA 之后，能立刻看到更短、更具体的运行证据，而不是先读长说明。
- `overview` 首屏更像 workspace 入口，而不是一张靠文案解释自身的控制台封面。

## 本轮结果

  - flow 阶段序列

- 移动端把 flow 阶段改成横向可滑的紧凑卡片，首屏先看到运行证据，再往下进入焦点区。
