# 7Flows 开源与商业策略基线

> 文档分层约定：本文只保留当前仍成立、且对产品定位、版本分层、授权边界和协作开发有直接指导作用的策略基线。详细推导过程、内部讨论素材和传播文案草稿如需本地保留，请放在 `docs/.taichuy/`；当前实现事实仍以 `docs/dev/runtime-foundation.md` 为准。

## 1. 一句话定位

7Flows 是以 OpenClaw / 本地 AI 助手“黑盒变透明”为传播切口、以 agent workflow kernel 为产品内核、向组织级治理能力延展的可视化执行与控制面平台。

## 2. 双层定位

### 2.1 对外传播与市场切口

- 短期优先围绕 OpenClaw / 本地 AI 助手“执行过程看不清、失败点难定位、工具调用难追踪”的痛点展开。
- README、官网、demo 和冷启动案例优先讲清楚执行透明、trace / replay、错误定位和工具调用追踪，而不是先讲大而全平台愿景。
- 首轮传播对象优先聚焦独立开发者、极客、小团队技术负责人、自动化实践者和本地 AI 助手重度用户。

### 2.2 内部产品内核

- 内部产品内核仍是面向多 Agent 协作的 workflow 平台，而不是某个上游产品的 UI 包装层。
- 平台内部统一以 `7Flows IR`、Durable Runtime、发布网关、插件兼容层和统一事件流为事实中心。
- OpenClaw 集成边界仍保持 `workflow-backed provider`：OpenClaw 对接的是 7Flows 发布网关，而不是直接理解 7Flows 内部 DSL。

### 2.3 关键约束

- 外部叙事可以 OpenClaw-first，但内部架构不能退化成“OpenClaw 专属执行器”。
- 市场切口服务冷启动，不能反向主导 `7Flows IR`、runtime、发布协议和插件边界。

## 3. 核心战略原则

### 3.1 开源给协作，商业给治理

- 社区版负责 adoption、开发者入口、协作编排体验、协议沉淀和生态扩展。
- 商业版负责组织治理、责任边界、成本控制、部署保障和官方服务。
- 商业能力必须建立在同一 kernel 上，不允许另起执行引擎、另起流程语义或第二套内部 DSL。

### 3.2 社区版专注方向

- 社区版首先要把 OpenClaw / 本地 AI 助手的黑盒透明化、基础编排、基础追溯和基础协作做扎实。
- 社区版重点放在 `7Flows IR`、runtime、published surface、trace / replay、插件协议、自部署与社区扩展入口。
- 社区版必须是真实可用的协作入口，而不是演示版、阉割版或只用来导流的样板工程。
- 社区版短期不追求把组织治理、复杂审批、企业合规、重型 IAM、托管 SLA 和白标控制面一起塞进主仓库。

### 3.3 授权表达要诚实

- 如果仓库授权采用 Apache 2.0 基底 + 附加条件的 community license，就必须诚实写成“community / source-available 授权”，而不是继续标成纯 `MIT` 或纯 `Apache-2.0`。
- “开源给协作”强调的是源码开放协作入口和生态扩展路径，不等于必须放弃所有商业保留边界。
- 涉及授权、商用边界、前端品牌、白标、对立面服务或商业授权触发条件时，以 `LICENSE` 为准，不要只凭 README 文案判断。

### 3.4 中短期像产品，长期像平台

- 中短期优先打透真实痛点和最小可卖闭环。
- 长期护城河优先放在 `workflow / runtime / plugin / trace` 标准层和生态，而不是单点闭源功能。
- 任何短期实现都不应锁死未来的平台化边界。

## 4. 用户与付费对象

### 4.1 传播对象

- 独立开发者 / 极客
- 小团队技术负责人
- 自动化实践者、集成者、中转商

### 4.2 付费对象

- 需要多人协作治理的小团队负责人
- 企业内部 AI 平台团队
- 需要补组织边界、成本控制、责任链和服务保障的组织

### 4.3 双用户视角

- 7Flows 的用户不仅是人，也包括把 workflow 当作稳定能力源来消费的 AI。
- 社区版 / 商业版分层不能破坏“人机交互、人与 AI 协作、AI 独立操作”三种场景的结果语义一致性。

## 5. 版本分层边界

> 下列边界是目标设计，不等于当前仓库已经全部实现。

### 5.1 Community / Self-host

- `7Flows IR`、workflow schema、runtime、published surface、trace / replay 基础能力
- OpenClaw / 本地 AI 助手场景下的执行透明、工具调用追踪、基础回放和错误定位
- 可视化编排、基础自部署、插件协议、SDK、社区模板与社区扩展入口
- 面向真实 adoption 的基础协作能力
- 社区版专注“让个人和小团队先把 workflow 跑起来、看得清、调得动”，而不是先做重治理控制面

### 5.2 Team

- 多 workspace
- 发布治理、发布确认或基础审批
- 团队级使用报表、成本面板、风险告警
- 环境隔离
- 私有模板库和更长时间的治理型历史留存

### 5.3 Enterprise

- 组织级治理、审计日志、高级审批链
- 高级 RBAC / ABAC、SSO / SAML / SCIM
- 成本中心、预算 / 配额策略、模型 / 连接器白名单
- 私有节点仓库、私有部署与组织级责任链

### 5.4 Managed / Service

- 官方托管执行、日志、artifact、queue 等持续资源能力
- SLA、专属支持、升级保障、迁移服务
- 咨询、定制集成、企业导入等官方交付能力

## 6. 授权策略

### 6.1 当前授权形态

- 当前仓库授权以根目录 `LICENSE` 为准，采用 `7Flows Community License`。
- 该许可证以 Apache License 2.0 为基础，并额外保留商业化治理边界，因此它不是纯 `Apache-2.0`、也不是纯 `MIT`。
- 社区版强调源码开放协作、自部署和二次开发入口；商业版通过单独商业授权承接多租户托管、白标和治理控制面等场景。

### 6.2 社区版允许的主要场景

- 个人、团队或组织内部使用
- 单租户或客户专属部署
- 基于源码做定制化、自部署交付或与自有业务系统集成
- 作为自有应用或内部平台的 workflow / runtime 能力源使用

### 6.3 需要商业授权的主要场景

- 运营面向多个客户或多个独立 workspace 的共享托管服务、多租户控制面或平台型 SaaS
- 直接基于 7Flows 源码提供商业化对立面的 agent workflow 平台、控制面或托管服务
- 去除、替换或遮蔽前端中的 7Flows 标识、版权信息和可见归属，以白标方式分发或销售

### 6.4 前端与品牌边界

- 本仓库中的前端默认指 `web/` 目录及其构建产物。
- `LICENSE` 不授予 7Flows 名称、LOGO、品牌视觉或其他商标性资产的自由使用权，前端去标识或品牌替换应走商业授权。
- 如果后续存在独立设计资产、品牌规范或 UI 资源包，应继续与代码授权分层管理，不把品牌资产默认并入社区自由使用范围。

## 7. 路线判断

### 7.1 近 90 天

- 先把 OpenClaw / 本地 AI 助手的黑盒调试切口打透。
- 重点验证执行透明、trace / replay、错误定位、基础编排和基础协作是否足够形成 adoption。

### 7.2 近 180 天

- 在 Community adoption 基础上，定义并验证 Team 最小可卖闭环。
- 重点围绕多 workspace、发布治理、团队报表、告警、环境与基础成本能力。

### 7.3 1 年视角

- 在已有 adoption 和 Team 付费闭环基础上，再建立 Enterprise 叙事。
- 重点围绕组织治理、私有部署、合规与官方服务。

## 8. 对文档与协作的影响

### 8.1 README / 官网 / Demo

- 优先使用“黑盒变透明”的顺序表达：
  - 先讲痛点
  - 再讲执行透明、trace / replay、错误定位
  - 再讲团队协作与治理升级路径
  - 最后再讲长期平台愿景
- 需要同时写清楚社区版的专注方向和许可证边界，避免把“社区协作入口”误写成“多租户商用平台免费版”。

### 8.2 产品与技术基线

- `docs/product-design.md` 和 `docs/technical-design-supplement.md` 继续负责内部产品和技术边界。
- 涉及产品切口、版本分层、授权边界、前端品牌、商业授权触发条件时，先看本文和 `LICENSE`，再决定写入产品基线还是当前事实索引。

### 8.3 AI 协作开发

- AI 在开发、审查、重构、测试时，除阅读 `AGENTS.md`、产品/技术基线、当前事实外，也应在相关任务上优先加载 `.agents/skills/*/SKILL.md`。
- 若任务涉及 OpenClaw 场景、Community / Team / Enterprise 分层、授权边界、前端品牌或商业化切分，应同时参考本文与 `LICENSE`，避免只按技术局部最优推进。

## 9. 当前仍需继续明确的现实约束

- 当前仓库主要落地的是 Community kernel 与运行时基础，不应提前承诺已具备 Team / Enterprise 成熟控制面。
- 当前 `LICENSE` 已切换为 Apache 2.0 基底 + 附加条件的 community license；后续仍需继续明确“多租户”“商业化对立面”“前端品牌替换”和“商业授权触发条件”的执行口径。
- `organization / workspace / member / role / publish governance` 仍主要停留在目标设计，后续需要收敛成最小可实现领域模型。
