# 7Flows 开源与商业策略基线

> 文档分层约定：本文只保留当前仍成立、且对产品定位、版本分层、开源/商业边界和协作开发有直接指导作用的策略基线。详细推导过程、完整文案草稿和延展讨论保留在 `docs/7_flows_开源与商业双线计划书_v_1.md`；当前实现事实仍以 `docs/dev/runtime-foundation.md` 为准。

## 1. 一句话定位

7Flows 是面向 OpenClaw / 本地 AI 助手场景切入、面向 agent workflow 演进的开放执行底座与组织级控制面。

## 2. 双层定位

### 2.1 对外传播与市场切口

- 短期优先围绕 OpenClaw / 本地 AI 助手“黑盒执行看不清”的痛点展开。
- 首屏价值先讲清楚：执行步骤可见、工具调用可追踪、运行过程可回放、失败节点可定位。
- README、官网、demo 和冷启动案例优先服务开发者、极客、小团队技术负责人等传播对象。

### 2.2 内部产品内核

- 内部产品内核仍是面向多 Agent 协作的工作流平台，而不是某个上游产品的 UI 包装层。
- 平台内部统一以 `7Flows IR`、Durable Runtime、发布网关、插件兼容层和统一事件流为事实中心。
- OpenClaw 集成边界仍保持 `workflow-backed provider`：OpenClaw 对接的是 7Flows 发布网关，而不是直接理解 7Flows 内部 DSL。

### 2.3 关键约束

- 外部叙事可以 OpenClaw-first，但内部架构不能因此退化成“OpenClaw 专属执行器”。
- 市场切口服务冷启动，不能反向主导 `7Flows IR`、runtime、发布协议和插件边界。

## 3. 核心战略原则

### 3.1 开源做标准底座，商业做组织能力

- 开源版负责 adoption、开发者入口、协议沉淀和生态扩展。
- 商业版负责组织治理、责任边界、成本控制、部署保障和官方服务。
- 商业能力必须建立在同一 kernel 上，不允许另起执行引擎、另起流程语义或第二套内部 DSL。

### 3.2 开源给协作，商业给治理

- 开源版应该是“真实可用的协作入口”，不是演示品，也不应把“第二个用户”当作收费点。
- 商业版优先收费于多人后的治理复杂性，而不是基础采用门槛。
- 当前仓库仍处于 OSS kernel 和运行时基础建设阶段，Team / Enterprise / Managed 主要属于目标设计，不应冒充当前事实。

### 3.3 中短期像产品，长期像平台

- 中短期优先打透真实痛点和最小可卖闭环。
- 长期护城河优先放在 `workflow/runtime/plugin/trace` 标准层和开源生态，而不是单点闭源功能。
- 任何短期实现都不应锁死未来的平台化边界。

## 4. 用户与付费对象

### 4.1 传播对象

- 独立开发者 / 极客
- 小团队技术负责人
- 中转商、集成者、自动化实践者

### 4.2 付费对象

- 小团队负责人
- 企业内部 AI 平台团队
- 需要为内部自动化补治理、边界和服务保障的组织

### 4.3 双用户视角

- 7Flows 的用户不仅是人，也包括把 workflow 当作稳定能力源来消费的 AI。
- 开源 / 商业分层不能破坏“人机交互、人与 AI 协作、AI 独立操作”三种场景的结果语义一致性。

## 5. 版本分层边界

> 下列边界是目标设计，不等于当前仓库已经全部实现。

### 5.1 OSS / Community

- `7Flows IR` / workflow schema / runtime / published surface / trace-replay 基础能力
- 可视化编排、执行透明、工具调用追踪、基础回放、错误定位
- 插件协议、SDK、社区模板与社区扩展入口
- 自部署
- 面向真实 adoption 的基础协作能力

### 5.2 Team

- 多 workspace
- 发布治理、发布确认或基础审批
- 团队级使用报表、成本面板、风险告警
- 环境隔离
- 私有模板库和更长时间的治理型历史留存

### 5.3 Enterprise

- 组织级治理、审计日志、高级审批链
- 高级 RBAC / ABAC、SSO / SAML / SCIM
- 成本中心、预算 / 配额策略、模型 / 连接器白名单
- 私有节点仓库、私有部署与组织级责任链

### 5.4 Managed / Service

- 官方托管执行、日志、artifact、queue 等持续资源能力
- SLA、专属支持、升级保障、迁移服务
- 咨询、定制集成、企业导入等官方交付能力

## 6. 路线判断

### 6.1 近 90 天

- 先把 OpenClaw / 本地 AI 助手的黑盒调试切口打透。
- 重点验证执行透明、trace / replay、错误定位、基础编排和基础协作是否足够形成 adoption。

### 6.2 近 180 天

- 在 OSS adoption 基础上，定义并验证 Team 最小可卖闭环。
- 重点围绕多 workspace、发布治理、团队报表、告警、环境与基础成本能力。

### 6.3 1 年视角

- 在已有 adoption 和 Team 付费闭环基础上，再建立 Enterprise 叙事。
- 重点围绕组织治理、私有部署、合规与官方服务。

## 7. 对文档与协作的影响

### 7.1 README / 官网 / Demo

- 优先使用“黑盒变透明”的顺序表达：
  - 先讲痛点
  - 再讲执行透明、trace / replay、错误定位
  - 再讲团队协作与治理升级路径
  - 最后再讲长期平台愿景
- 不能在 README 里把 Team / Enterprise 目标能力写成当前仓库事实。

### 7.2 产品与技术基线

- `docs/product-design.md` 和 `docs/technical-design-supplement.md` 继续负责内部产品和技术边界。
- 涉及产品切口、版本分层、开源/商业边界时，先看本文，再决定写入产品基线还是当前事实索引。

### 7.3 AI 协作开发

- AI 在开发、审查、重构、测试时，除阅读 `AGENTS.md`、产品/技术基线、当前事实外，也应在相关任务上优先加载 `.agents/skills/*/SKILL.md`。
- 若任务涉及对外叙事、OpenClaw 场景、版本边界、治理能力或商业化切分，应同时参考本文，避免只按技术局部最优推进。

## 8. 当前仍需继续明确的现实约束

- 当前仓库主要落地的是 OSS kernel 与运行时基础，不应提前承诺已具备 Team / Enterprise 成熟控制面。
- 当前仓库许可证文件为 GPLv2 文本；如果后续坚持 Open Core，还需要单独明确开源版许可证、商业控制层分发边界和私有部署授权方式。
- 组织、成员、角色、多 workspace、审批流、审计与企业部署等能力当前仍以目标设计为主，后续需结合代码事实逐步落地。
