# 用户认证与团队接入后端落地边界

## 时间

`2026-04-12 17`

## 谁在做什么

- 用户要求将 `docs/superpowers/specs/1flowse/modules/01-user-auth-and-team` 对应后端直接落地，并明确要求同时覆盖模块 `02 权限与资源授权` 的角色 CRUD 与权限绑定接口。
- AI 负责在现有 Rust 后端骨架上补齐认证、团队、成员、角色、权限、审计、初始化与测试。

## 为什么这样做

- 当前后端仍是空骨架，若只落模块 `01` 而不把角色与权限编辑链路一起打通，后续模块 `02` 会反过来改表、改接口、改会话载体，返工成本高。
- 用户偏好更激进的实现方式，希望在单轮开发中把主干能力整体写出来，再通过验证结果逐项修正。

## 为什么要做

- 先把控制台认证、团队与权限主干打牢，后续工作台、应用、页面、发布与状态模块都可以直接复用统一的成员、角色、权限与会话基础设施。
- 提前固定数据模型、审计边界与权限目录，减少后续跨模块重复收敛。

## 截止日期

- 未指定

## 决策背后动机

- 本轮范围扩展为：模块 `01` 全闭环 + 模块 `02` 的角色 CRUD 与权限绑定接口一并实现。
- 权限目录一次性按已确认资源范围种全，不只种当前接口所需的最小子集。
- 用户/成员模型除业务字段外默认继承基类字段：`id`、`introduction`、`created_by`、`created_at`、`updated_by`、`updated_at`。
- 用户相关字段明确包含：`account`、`email`、`phone`、`avatar_url`、`password`、`default_display_role`、`name`、`nickname`。
- 登录标识以 `account` 为主；默认允许 `email` 直接作为登录标识；`phone` 为非必填且默认不启用登录；后端需支持按用户配置决定 `email/phone` 是否可作为登录账号。
- 团队 Logo 本轮先存 `logo_url` 字符串，不提前引入对象存储绑定。
- `api-server` 启动时自动执行幂等 bootstrap：按 `.env` 创建缺失的团队、`root` 和内置角色；已存在数据不覆盖；后续 `.env` 改密码不自动同步，仅支持脚本重置。
- `root` 账号出现在成员列表中，但不能被禁用、删除、解绑 `root` 或改成其他角色；`root` 可自行修改昵称与头像；密码只允许本人修改或走兜底脚本。
- 会话策略允许多端登录；账号被禁用或被管理员重置密码时，强制失效该用户全部会话；用户自行修改密码成功后，全部会话失效并重新登录。
- 后台重置密码不强制用户首次登录立即改密。
- `root` 默认以 `root` 视角进入，同时允许切为 `admin/manager` 视角做前端裁剪测试；普通用户默认展示角色失效时自动回退到首个有效角色。
- 本轮补最小审计表/事件，至少覆盖：成员创建、账号禁用、密码重置、角色绑定变更、团队配置变更。
