# User Preferences

这里记录在开发过程中确认的“长期有效用户偏好”，用于指导后续实现、文档和 skill 调整。

## 使用原则

- 只记录稳定偏好，不记录一次性任务要求。
- 偏好需要能指导未来决策，而不是仅复述聊天内容。
- 如果某条偏好已经沉淀为仓库级规则或专项 skill 规则，需要在“落地动作”里说明同步位置。

## 记录模板

```md
## YYYY-MM-DD 偏好主题

- 偏好：
- 来源：
- 影响范围：
- 落地动作：
- 备注：
```

## 当前已记录偏好

## 2026-03-14 节点执行采用分级执行而不是全量沙箱化

- 偏好：7Flows 的节点执行架构应采用“统一工作流执行器 + 分级节点执行层 + 少量高风险节点强隔离”的方式，而不是把所有节点一刀切放进沙箱；默认让绝大多数可信节点在 worker 内轻量执行，把 `sandbox` / `microvm` 留给代码节点、自定义节点、插件节点和高风险工具节点。
- 来源：用户在本轮关于节点与沙箱设计的架构讨论中明确认可“更像 n8n 的工作流引擎主控 + 更像 Dify 的高风险节点隔离”这条工程路线，并要求把该方向正式更新到仓库规则和设计文档中。
- 影响范围：`AGENTS.md`、`docs/product-design.md`、`docs/technical-design-supplement.md`、`docs/dev/runtime-foundation.md`，以及后续 `runtimePolicy.execution`、Execution Adapter Registry、Tool Gateway、sandbox adapter、插件风险分级与前端节点配置表单设计。
- 落地动作：本轮已把“NodeType 与 execution class 分离”“工作流执行器唯一主控”“不做所有节点默认重沙箱化”的规则同步到仓库级协作约定、产品设计、技术补充、当前事实索引和开发记录中，作为后续运行时演进的明确方向。
- 备注：这里强调的是“按风险和能力分级执行”，不是削弱沙盒地位；沙盒仍是核心能力，但应聚焦在真正高风险、需隔离的节点和工具上。

## 2026-03-14 安全基座先做分级访问管理而不是业务分类

- 偏好：7Flows 的安全与授权基座先做统一的分级访问管理流程，只管理 `sensitivity_level`、访问请求、人工审核、通知和审计闭环；具体资源内容和业务语义由应用厂商侧提供，不要在平台里预置行业分类。
- 来源：用户在本轮架构讨论中明确要求，把“授权=人工审核，人工审核需要通知；平台只做分级管理，不做业务分类”沉淀为长期方向。
- 影响范围：`AGENTS.md`、`docs/product-design.md`、`docs/technical-design-supplement.md`、`docs/dev/runtime-foundation.md`，以及后续 Tool Gateway、审批流、通知适配器、运行追溯与前端治理视图设计。
- 落地动作：本轮已把该方向同步到产品设计、技术补充、当前事实索引、开发记录与仓库级架构边界，后续实现将优先落“统一敏感访问控制能力”，而不是先做行业分类体系或独立人工审核节点。
- 备注：这里强调的是“平台提供流程与治理能力”，不是让平台理解每个业务字段的行业含义；业务侧只需声明资源文本与敏感等级即可接入规则。

## 2026-03-10 文件体量控制

- 偏好：单个代码文件不应无限增长；后端代码文件原则上不超过 1500 行，前端代码文件原则上不超过 2000 行，超过时要优先评估拆分、抽离和封装，而不是继续堆叠职责。
- 来源：用户在“进行下一步开发”过程中明确提出，希望把文件行数上限沉淀为长期开发偏好。
- 影响范围：`api/`、`web/`，以及对应的 review、refactor、testing 和开发记录。
- 落地动作：已记录到 `docs/dev/user-preferences.md`；后续新增实现、代码审查和组件/模块重构时，将把文件体量作为显式检查项，并在接近或超过阈值时优先建议拆分。
- 备注：这是工程可维护性约束，不是机械性的硬性阻断规则；如果短期内因迁移或集中改造暂时超过阈值，需要尽快补上拆分计划。

## 2026-03-10 工具读取中文内容默认使用 UTF-8

- 偏好：使用终端或工具读取包含中文的文件时，应默认显式使用 UTF-8，避免因编码错误看到乱码后再做错误判断或修改。
- 来源：用户指出多次工具读取未显式使用 UTF-8，导致中文内容出现乱码。
- 影响范围：文档读取、代码查看、审查、开发记录维护，以及所有需要通过工具读取中文内容的流程。
- 落地动作：后续读取中文文档或源码时优先显式使用 UTF-8；同时将该规则同步到仓库级协作约定。
- 备注：如果文件编码不是 UTF-8，应先确认实际编码，再继续分析或修改。
