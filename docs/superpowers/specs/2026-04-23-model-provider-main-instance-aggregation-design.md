# 模型供应商主实例聚合设计

## 背景

当前模型供应商的“主实例”语义是“从某个 `ready` 子实例里选一个作为运行时解析目标”。这会带来两个问题：

1. 页面上的“主实例”更像是在选一个真实实例，而不是在管理该供应商下的统一模型入口。
2. 节点侧只保存 `provider_code + model_id`，运行时再通过 `primary_instance_id` 反查真实实例，无法表达“同一供应商下多个实例都提供同名模型”的来源差异。

这与产品目标不一致。目标中的“主实例”应该是一个固定存在的聚合视图，用来汇总该供应商下已接入子实例的模型；真实执行仍然发生在某个子实例上。

## 目标

本次设计将“主实例”重定义为“供应商级虚拟聚合实例”，满足以下要求：

1. 每个供应商固定存在一个主实例，用户不再手动从子实例中选“谁是主实例”。
2. 主实例只聚合子实例，不是一个可单独配置密钥的真实实例。
3. 子实例只支持按“整实例”接入主实例，不支持按单个模型粒度接入。
4. 主实例中的模型列表按来源实例分组展示。
5. LLM 节点保存 `provider_code + source_instance_id + model_id`，执行时直接命中对应子实例。
6. 新建子实例默认自动接入主实例，这个默认规则在供应商级可关闭，实例级仍可单独调整。

## 非目标

本次不做以下内容：

1. 不兼容旧的 `manual_primary` 语义，不做历史迁移和回退逻辑。
2. 不支持“只接入某个实例的部分模型”。
3. 不在 LLM 选择器中显式把实例名拼到每个模型标签上，只保留分组。
4. 不在实例被移出主实例后自动修复已有节点；节点值保留，报错交由管理员处理。

## 术语

### 主实例

供应商级固定存在的虚拟聚合视图。它不保存密钥，不直接执行请求，只暴露“当前接入的子实例模型池”。

### 子实例

真实的模型供应商实例，拥有自己的配置、密钥、候选模型缓存和启用模型列表，运行时请求最终落到这里。

### 接入主实例

指某个子实例被标记为“included_in_main = true”。接入后，该子实例的启用模型会被主实例实时聚合；移出后，主实例不再暴露该实例的模型。

## 方案选择

本设计采用“方案 C：主实例是虚拟聚合实例，子实例是真实执行实例”。

理由：

1. 概念稳定。主实例负责聚合，子实例负责执行，职责清晰。
2. 交互自然。用户不再需要理解“为什么要从子实例里挑一个当主实例”。
3. 运行时准确。节点保存来源实例后，即使不同子实例存在同名模型，也能明确落到正确的真实实例。
4. 扩展性更好。后续如果需要做主实例摘要、模型分组、来源过滤，语义边界已经明确。

## 领域模型调整

### 供应商级聚合配置

为每个 `provider_code` 引入供应商级主实例配置，至少包含：

1. `provider_code`
2. `auto_include_new_instances: boolean`

语义：

1. `auto_include_new_instances = true` 时，新建子实例默认接入主实例。
2. `auto_include_new_instances = false` 时，新建子实例默认不接入主实例。
3. 该配置只决定新建默认值，不覆盖已有子实例的实例级选择。

### 子实例新增字段

在真实子实例记录上新增：

1. `included_in_main: boolean`

语义：

1. 只有 `included_in_main = true` 的子实例才会被主实例聚合。
2. 主实例实际暴露模型时，还需要额外满足子实例处于 `ready` 状态。

### 移除旧路由语义

废弃当前以 `primary_instance_id` 为核心的供应商路由配置。

旧语义：

1. `routing_mode = manual_primary`
2. `primary_instance_id`

新语义：

1. 主实例是固定存在的供应商级聚合视图，不再需要从子实例中选主实例。
2. 节点执行目标不再通过“供应商 -> primary_instance_id”二次解析，而是直接依赖节点上记录的 `source_instance_id`。

## 主实例聚合规则

主实例的模型池按实时派生生成，不做快照复制。

### 聚合输入

对于某个供应商，主实例聚合输入为：

1. 同 `provider_code` 的全部子实例
2. 过滤条件：
   - `included_in_main = true`
   - `status = ready`

### 聚合输出

主实例暴露的模型结构为：

1. 按来源实例分组
2. 每组展示该实例当前启用模型，即 `enabled_model_ids`
3. 同名模型允许在不同实例分组下重复出现

### 实时同步行为

以下变更会实时影响主实例聚合结果：

1. 子实例 `included_in_main` 开关变化
2. 子实例 `status` 变化
3. 子实例 `enabled_model_ids` 变化
4. 子实例 `display_name` 变化
5. 子实例删除

## 前端交互设计

### 供应商列表（L0）

供应商列表只负责概览，不再承载“从哪个子实例里选主实例”的动作。

列表摘要应包含：

1. 实例总数
2. 已接入主实例的子实例数量
3. 主实例聚合模型数摘要
4. 供应商级默认规则摘要，例如“新实例默认加入主实例：开/关”

### 供应商实例弹窗（L1）

供应商实例弹窗结构调整为：

1. 顶部固定“主实例”区块
2. 下方为“子实例列表”

#### 主实例区块

展示：

1. 主实例说明文案，明确其为聚合视图
2. 供应商级默认开关：`新实例默认加入主实例`
3. 主实例当前接入的子实例数量
4. 主实例当前聚合模型分组

模型分组展示形式：

```text
百炼
- qwen-plus
- qwen3.5-plus

百炼2
- qwen-max
```

不再出现“主实例”下拉框。

#### 子实例列表

每个子实例项展示：

1. 实例名称
2. 状态
3. 生效模型数
4. 缓存模型数
5. Base URL 等摘要信息

每个子实例项提供：

1. `加入主实例` 开关
2. 编辑
3. 刷新候选模型
4. 刷新模型
5. 删除实例

删除旧的“主实例”标签，不再强调某个真实实例被选为主实例。

### 实例 Drawer（L2）

创建 / 编辑实例 Drawer 继续负责真实子实例配置。

新增字段：

1. `加入主实例`

默认值来源：

1. 新建实例时，默认值取供应商级 `auto_include_new_instances`
2. 编辑实例时，回填当前 `included_in_main`

### LLM 节点模型选择器

模型选择器改为读取“供应商主实例聚合结果”，并按实例分组显示模型。

选择器行为：

1. 先选择供应商
2. 再在该供应商下按来源实例分组展示模型
3. 分组标题为实例名称
4. 单个模型项只显示模型名称，不拼接实例名

选中后，节点值写入：

1. `provider_code`
2. `source_instance_id`
3. `model_id`

## API 合同调整

### 供应商实例列表

实例列表响应中的每个子实例需要包含：

1. `included_in_main`

移除或废弃：

1. `is_primary`

### 供应商级主实例配置

新增供应商级接口，用于读取和更新：

1. `auto_include_new_instances`

该接口只作用于供应商级默认规则，不再承载 `primary_instance_id` 更新。

### 模型供应商选项接口

当前 `model-provider options` 接口返回：

1. `effective_instance_id`
2. `effective_instance_display_name`
3. `models`

新接口语义改为返回主实例聚合视图，至少包含：

1. `provider_code`
2. `display_name`
3. `main_instance`
   - `provider_code`
   - `auto_include_new_instances`
   - `group_count`
   - `model_count`
4. `model_groups`
   - `source_instance_id`
   - `source_instance_display_name`
   - `models`

说明：

1. `model_groups` 是 LLM 选择器的直接数据源。
2. 不再用 `effective_instance_id` 表达单一真实实例。

## 节点配置合同

### 新合同

LLM 节点 `config.model_provider` 改为：

```json
{
  "provider_code": "openai_compatible",
  "source_instance_id": "instance-123",
  "model_id": "gpt-4o-mini"
}
```

### 校验规则

节点校验时需要检查：

1. `provider_code` 必填
2. `source_instance_id` 必填
3. `model_id` 必填
4. `source_instance_id` 对应实例必须存在
5. 该实例必须属于 `provider_code`
6. 该实例必须处于 `ready`
7. 该实例必须当前接入主实例
8. `model_id` 必须出现在该实例的当前可用模型内

### 失效保留策略

如果节点原先引用的实例后来被移出主实例，或者该模型不再可用：

1. 不自动清空节点当前值
2. 不自动改写为别的实例或别的模型
3. 在编辑器校验和运行前校验中报错
4. 由管理员手动调整

## 运行时设计

### 编译上下文

运行时编译不再围绕“供应商 -> primary_instance_id”构建上下文。

改为：

1. 编译上下文仍可按供应商组织可用模型视图，用于后端编译期校验与运行前校验
2. 真正执行时，以节点上的 `source_instance_id` 为准

前端模型选择器不依赖运行时编译上下文，继续通过 `model-provider options` 接口读取主实例聚合结果。

### 执行解析

执行 LLM 节点时：

1. 从节点配置读取 `source_instance_id`
2. 加载对应真实子实例
3. 校验供应商、状态和模型可用性
4. 使用该子实例的密钥与连接配置发起请求

这样可避免“两个子实例同名模型却只能靠供应商兜底解析”的歧义。

## 页面文案原则

为避免继续混淆“主实例”和“真实实例”，页面文案遵循以下规则：

1. “主实例”只表示供应商级聚合视图
2. “子实例”或“实例”表示真实可执行实例
3. 不再出现“选择某个实例作为主实例”的文案
4. 在主实例区块补充说明：“主实例会聚合已接入实例的模型，不直接保存密钥或执行请求”

## 测试要求

### 后端

至少覆盖：

1. 新建实例时，`included_in_main` 默认值继承供应商级 `auto_include_new_instances`
2. 更新实例 `included_in_main` 后，主实例聚合结果实时变化
3. 模型供应商选项接口按实例分组返回模型
4. 同名模型可出现在不同实例分组中
5. 运行时按节点 `source_instance_id` 解析真实实例
6. 实例被移出主实例后，已有节点保留旧值但校验失败

### 前端

至少覆盖：

1. 供应商实例弹窗不再显示“主实例”选择下拉框
2. 主实例区块始终存在
3. 子实例的“加入主实例”开关能影响主实例分组展示
4. 新建实例时默认值取供应商级默认规则
5. LLM 模型选择器按实例分组展示模型
6. 选中模型后写入 `provider_code + source_instance_id + model_id`
7. 节点引用失效时保留旧值并展示校验错误

## 风险与取舍

### 风险 1：合同改动面较大

会同时影响：

1. 设置页实例管理
2. 模型供应商 options 接口
3. LLM 节点字段合同
4. 运行时编译与校验

处理方式：

1. 直接按开发初期心态整体切换，不做兼容分支。

### 风险 2：主实例和子实例概念容易再次混淆

处理方式：

1. 严格区分文案
2. 严格区分 API 结构
3. 严格禁止把主实例当成真实实例保存密钥或执行请求

### 风险 3：失效保留策略会暴露更多运行错误

处理方式：

1. 接受这是产品上的明确取舍
2. 用校验错误把问题暴露给管理员，而不是在系统内部做隐式替换

## 结论

本次将模型供应商的“主实例”重构为“供应商级虚拟聚合实例”，把真实执行能力留在子实例上。系统不再通过单一 `primary_instance_id` 解析模型，而是通过“主实例聚合暴露选择入口 + 节点记录来源实例”来完成精确选择和执行。

这能同时解决当前设计中的三个核心问题：

1. 主实例语义不清
2. 多实例同名模型无法表达来源
3. 设置页和运行时的概念不一致
