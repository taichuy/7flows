# Workflow Quick Add Hover Preview 设计说明

## 1. 背景

当前 workflow editor 画布上的 quick-add 面板由 `web/components/workflow-editor-workbench/workflow-canvas-quick-add.tsx` 渲染。现有实现中，每个候选项都会直接显示名称、meta 和 description，导致：

- 列表项高度过大，候选项稍多就迅速把面板撑长。
- 用户在浏览节点时，需要跨越大量重复文本，搜索和扫读效率低。
- 与当前希望参考的 Dify 风格不一致：默认应优先展示可快速扫描的节点名称，详细信息按需查看。

本次改造目标是把 quick-add 面板收口为更紧凑的“名称列表 + 悬浮预览”选择器，同时保持现有节点插入主链能力、搜索能力和分类能力不变。

## 2. 目标与非目标

### 2.1 目标

- 降低 quick-add 面板默认高度占用，让单列列表能在同样画布高度下承载更多节点。
- 默认只展示节点名称，减少重复文案对扫读的干扰。
- 鼠标悬浮到节点名称时，在侧边固定预览区展示该节点的名称、类型和描述。
- 保留现有 `节点 / 工具` tab、搜索框、点击插入节点的主流程。
- 兼容键盘 focus 场景，让 hover preview 不只依赖鼠标。

### 2.2 非目标

- 不调整 node catalog 数据结构，不新增后端字段。
- 不改变 quick-add 的插入语义，不修改节点自动连线逻辑。
- 不在本轮引入图标系统、富文本描述或新的节点分组规则。
- 不重做整个 editor sidebar / inspector 布局。

## 3. 选定方案

采用“两栏 quick-add 菜单”：

- 左侧为紧凑列表，仅显示节点名称。
- 右侧为固定预览区，显示当前悬浮或 focus 的节点详细信息。
- 左侧列表独立滚动，右侧预览区不跟随列表滚动。

未采用的方案：

- 外挂浮层预览：容易出现 hover 抖动和边缘溢出。
- tooltip / popover 预览：预览面积过小，不适合承载多行介绍。

## 4. 交互设计

### 4.1 默认状态

- 面板打开后，默认选中当前 tab 下过滤结果的第一项。
- 左侧列表项默认只显示 `label`。
- 右侧预览区默认显示当前默认项的：
  - 节点名称
  - 节点类型 / capability group 摘要
  - description

### 4.2 悬浮与 focus

- 鼠标移入某个列表项时，更新右侧预览区内容。
- 键盘 focus 到某个列表项时，同样更新右侧预览区内容。
- 鼠标移出列表项但仍在面板内时，保留最后一次命中的预览内容，不闪回空态。

### 4.3 点击插入

- 点击任一列表项后，沿用现有 `onQuickAdd(type)` 行为。
- 插入成功后关闭菜单，并清空当前搜索关键字。

### 4.4 搜索与 tab 切换

- 搜索逻辑保持现状，仍按 `label + description + type + capabilityGroup` 过滤。
- 搜索结果变化后，如果当前预览项仍存在，则保留。
- 若当前预览项已不在过滤结果中，则自动回退到过滤结果第一项。
- 切换 `节点 / 工具` tab 时，预览项重置为新 tab 的第一项。

### 4.5 空态

- 若过滤结果为空，左侧列表隐藏，右侧预览区隐藏，保留现有“没有匹配的可插入节点”空态文案。

## 5. 布局与视觉

### 5.1 菜单尺寸

- quick-add 菜单整体宽度从当前单列窄菜单扩展为双栏宽度。
- 最大高度继续受画布 / 视口限制，但应比当前更稳定地贴近可视区高度。
- 左侧列表负责滚动；右侧预览区固定，不与列表内容一起滚动。

### 5.2 列表项

- 每项压缩为单行名称主文案。
- 保留 hover / active / focus 态，用于明确当前预览项来源。
- 不在默认列表项中显示 description 或额外说明文案。

### 5.3 预览区

- 右侧使用卡片式信息块，至少包含标题、meta 和描述。
- 如果 description 为空，显示已有的兜底文案，而不是留空。
- 预览区风格与现有 editor 浅色画布面板保持一致，不单独引入新主题。

### 5.4 响应式

- 桌面和普通编辑器宽度下默认使用双栏。
- 在菜单宽度明显不足的场景下，退化为单栏列表 + 底部预览区，避免预览压坏列表宽度。

## 6. 实现边界

主要改动限定在前端 quick-add 层：

- `web/components/workflow-editor-workbench/workflow-canvas-quick-add.tsx`
- `web/app/globals.css`
- quick-add 相关测试

实现原则：

- 继续复用现有 `WorkflowCanvasQuickAddOption` 数据结构。
- 通过组件内部状态维护当前 preview item，不向上游 graph / shell 泄漏新的编辑器状态。
- 不为本次 hover preview 预留额外抽象层，避免过度设计。

## 7. 可访问性

- 列表项保持 button 语义。
- hover 之外，focus 也必须能驱动右侧预览区更新。
- 预览区内容变更不依赖动画才能理解。
- 保留 Escape 关闭、点击外部关闭等现有菜单交互。

## 8. 测试与验证

至少补充或扩展以下验证：

- quick-add 打开后默认展示第一项预览。
- hover 某个列表项时，预览区切换到对应节点。
- focus 某个列表项时，预览区同样切换。
- 搜索结果变化导致当前预览项失效时，会回退到首个结果。
- 点击列表项后触发插入并关闭菜单。
- 无匹配结果时显示空态。

实现完成后，按前端基线至少运行：

- `cd web && pnpm lint`
- 与本次组件相关的 targeted test

## 9. 风险与控制

- 风险：菜单宽度增加后可能遮挡更多画布区域。
  控制：限制最大宽度，并在小屏下回退为单栏 + 底部预览。

- 风险：hover 与 click 状态混用造成预览闪烁。
  控制：采用“最后一次有效命中保留”的策略，不在 `mouseleave` 时主动清空预览。

- 风险：搜索 / tab 切换后预览项引用失效。
  控制：在过滤结果变化时做一次存在性校验，不存在就回退到首项。

## 10. 交付结果

交付后，quick-add 面板应满足以下体验：

- 默认看到的是紧凑的节点名称列表，而不是长段说明。
- 用户悬浮到任一节点名称时，右侧可立即查看详细介绍。
- 列表变长时，主要由左侧列表滚动承担，不再把整块菜单无限拉长。
- 插入节点主链行为与当前实现保持一致。
