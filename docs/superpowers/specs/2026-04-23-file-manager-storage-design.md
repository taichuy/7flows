# 文件管理与对象存储分层设计

## 背景

当前仓库里真正落盘到文件系统的内容主要是插件安装产物，目录和生命周期都围绕 `api/plugins` 设计。这套机制不适合承接后续业务文件，尤其是：

1. 对话上传文件
2. 多模态模型输入附件
3. 图片生成等模型输出产物

这些文件与插件产物在权限、生命周期、引用关系和后续扩展方向上都不是同一类问题。如果继续复用插件目录，会很快把“业务文件”和“宿主运行产物”混在一起。

用户已经明确：

1. 业务文件体系要参考 `NocoBase` 的“文件记录表 + 存储器配置”分离架构
2. 官方首批只支持 `本地文件` 和 `rustFS`
3. `rustFS` 驱动默认内建，但不默认启用，用户自行在后台配置
4. 后续 `S3`、阿里对象存储、腾讯云对象存储等走同一扩展口
5. 文件记录要统一管理，用户上传文件和模型生成产物都进入统一文件体系
6. 第一版采用“真实多文件表 + 存储器绑定”，不是单表逻辑分组

## 目标

本次设计的目标是：

1. 为业务文件建立独立于插件产物的文件管理体系
2. 提供 `文件记录` 与 `存储器` 分离的稳定架构
3. 支持系统内建默认文件表 `attachments`
4. 支持 `workspace` 按固定模板新建自己的真实文件表
5. 支持文件表绑定文件管理器，并允许后续切换绑定
6. 保证文件表改绑存储器时只影响新上传文件，不影响旧文件读取
7. 让文件记录权限直接复用现有动态建表运行态 CRUD 权限
8. 为后续新增官方或插件化存储驱动预留统一扩展口

## 非目标

第一版不做以下内容：

1. 不做浏览器直传对象存储
2. 不做大文件分片上传
3. 不做历史文件迁移任务
4. 不做文件表自定义扩展字段
5. 不做文件去重、秒传、校验和治理
6. 不做文件版本管理
7. 不把业务文件接入插件安装目录

## 术语

### 存储器

指一个具体的文件落盘实现实例，例如：

1. 本地文件系统实例
2. `rustFS` 实例
3. 未来的 `S3`、`OSS`、`COS` 实例

### 文件表

指一张真实存在的文件业务表。它有固定模板字段，并在控制面注册为可用文件表。业务侧上传文件时，总是先选定文件表，再按该文件表当前绑定的存储器执行上传。

### 文件记录

指文件表中的一条记录，保存文件元数据和实际使用的 `storage_id` 快照。文件记录是业务侧唯一可靠的文件引用对象。

### 文件管理器

本设计里“文件管理器”就是文件表当前绑定的存储器实例。一个文件表任一时刻只能绑定一个当前存储器，但表内历史文件记录可以来自不同 `storage_id`。

## 方案选择

本设计采用“真实多文件表 + 存储器绑定”的方案。

理由：

1. 它最符合用户希望参考 `NocoBase` 分离架构的要求
2. 每个文件表都可以作为一等业务资源存在，而不是单表上的逻辑标签
3. `workspace` 后续可以有自己的文件表，而不需要把所有文件都挤进一个全局表
4. 文件表级绑定存储器后，控制面和业务心智都更稳定
5. 以后新增手动迁移任务时，也能自然地从“表当前绑定”和“记录历史存储器”两层语义演进

不采用“单核心文件资产表 + 逻辑文件表”的原因是：

1. 它实现上更简单，但不符合“支持新建真实文件表”的目标
2. 业务侧最终还是会把逻辑文件表理解成真实表，后续重构成本更高

## 总体架构

整体拆成三层：

1. `storage-object`
   负责文件存储驱动抽象和驱动注册
2. `control-plane`
   负责存储器、文件表、绑定关系和上传编排
3. 运行态真实文件表
   负责承接文件记录 CRUD 和业务关联

对应目录职责建议如下：

1. `api/crates/storage-object`
   - 定义 `FileStorageDriver` 抽象
   - 注册内建 `local` 和 `rustfs` 驱动
   - 负责 `put/delete/open_read/generate_access_url/healthcheck`
2. `api/crates/control-plane/src/file_management`
   - 负责存储器服务
   - 负责文件表注册服务
   - 负责上传编排和绑定规则
3. `api/apps/api-server/src/routes`
   - 暴露存储器管理、文件表管理和上传接口
4. 动态建表现有链路
   - 负责真实文件表创建、字段模板下发、运行态 CRUD 权限复用

## 作用域与权限边界

### 存储器作用域

存储器固定为 `system/root` 级资源：

1. 只有 `root` 可以创建、更新、启用、禁用和删除存储器
2. `workspace` 无法修改存储器配置
3. `workspace` 也不能直接决定文件表绑定到哪个存储器

### 文件表作用域

文件表分两类：

1. `system` 级默认文件表
   - 系统内建默认表 `attachments`
   - 归 `root/system` 管理
2. `workspace` 级文件表
   - 由某个 `workspace` 创建
   - 物理上是真实新表
   - 模板固定复用 `attachments`

### 绑定关系权限

文件表与存储器的绑定关系只允许 `root` 修改：

1. `root` 可以修改任意文件表的当前 `bound_storage_id`
2. `workspace` 只能使用自己可见的文件表
3. `workspace` 可以创建自己的文件表，但创建后默认绑定到系统默认存储器

### 文件记录权限

文件记录不发明新权限模型，直接复用现有运行态数据权限：

1. `create`
2. `view`
3. `edit`
4. `delete`
5. `own`
6. `all`

也就是说，文件表一旦创建完成，它的数据 CRUD 权限就按现有动态建表运行态口径执行，而不是额外发明一套“文件专用 ACL”。

## 核心数据模型

### `file_storages`

这是控制面系统表，用来记录可用存储器实例。

建议字段：

1. `id`
2. `code`
3. `title`
4. `driver_type`
   - `local`
   - `rustfs`
5. `enabled`
6. `is_default`
7. `config_json`
8. `rule_json`
9. `health_status`
10. `last_health_error`
11. `created_at`
12. `updated_at`
13. `created_by`
14. `updated_by`

说明：

1. `config_json` 保存驱动配置，例如本地根目录、`rustFS` endpoint、bucket 或 namespace 等
2. `rule_json` 保存大小、`mimetype`、命名策略等文件规则
3. `is_default` 只允许同时存在一个默认存储器

### `file_tables`

这是控制面系统表，用来注册系统里有哪些文件表，以及它们当前绑定哪个存储器。

建议字段：

1. `id`
2. `code`
3. `title`
4. `scope_kind`
   - `system`
   - `workspace`
5. `scope_id`
6. `model_definition_id`
7. `bound_storage_id`
8. `is_builtin`
9. `is_default`
10. `status`
11. `created_at`
12. `updated_at`
13. `created_by`
14. `updated_by`

说明：

1. `model_definition_id` 指向当前真实文件表对应的动态模型定义
2. `bound_storage_id` 是该文件表“当前用于新上传”的存储器
3. `attachments` 是系统内建默认记录，`is_builtin = true`

### 真实文件表模板

第一版所有真实文件表统一复用 `attachments` 模板，不开放扩展字段。

建议模板字段：

1. `title`
2. `filename`
3. `extname`
4. `size`
5. `mimetype`
6. `path`
7. `meta`
8. `url`
9. `storage_id`

再叠加运行态模型已有基础字段：

1. `id`
2. `created_at`
3. `updated_at`
4. `created_by`
5. `updated_by`

关键约束：

1. `storage_id` 是文件记录实际使用的存储器快照，不等于文件表当前绑定
2. `url` 只是派生或缓存字段，不是唯一真相
3. 真正可靠的读定位依据是 `storage_id + path + filename`

## 默认文件表与新建文件表

### 默认文件表

系统启动或迁移时自动确保存在一个默认文件表：

1. `code = attachments`
2. `scope_kind = system`
3. `is_builtin = true`
4. `is_default = true`

它默认绑定到系统默认存储器，第一版默认就是本地存储器。

### 新建文件表

`workspace` 新建文件表时，平台执行以下动作：

1. 走现有动态建表链路创建一张真实模型和真实物理表
2. 固定下发 `attachments` 模板字段
3. 在 `file_tables` 里创建注册记录
4. 自动绑定到当前系统默认存储器

第一版不支持：

1. 用户自己加字段
2. 用户自己删模板字段
3. 用户自己改绑定存储器

## 存储驱动抽象

在 `storage-object` 中定义统一驱动抽象：

1. `validate_config`
2. `healthcheck`
3. `put_object`
4. `delete_object`
5. `open_read`
6. `generate_access_url`

### `local` 驱动

官方内建，默认启用，默认创建一个系统默认实例。

职责：

1. 写入本地文件系统目录
2. 读取本地文件流
3. 生成内部或可访问 URL

### `rustfs` 驱动

官方内建，但默认不启用、不默认创建实例。

职责：

1. 调用 `rustFS` HTTP API 或 SDK 上传对象
2. 删除对象
3. 读取对象流或生成访问地址
4. 提供健康检查和配置校验

### 后续扩展

后续新增 `S3`、阿里云对象存储、腾讯云对象存储时：

1. 只接入 `storage-object` 的 driver registry
2. 不修改业务文件表结构
3. 不修改文件记录引用模型

## 上传链路

第一版统一采用：

`Browser -> API Server -> Storage Driver`

不做浏览器直传。

### 上传流程

1. 前端选择目标文件表
2. 后端根据 `file_table_id` 加载文件表注册记录
3. 读取该文件表当前 `bound_storage_id`
4. 加载对应存储器实例和驱动
5. 按存储器规则校验文件大小、类型和命名策略
6. 调用驱动写入对象
7. 得到 `path/filename/url/meta`
8. 在目标真实文件表中插入文件记录，并写入实际 `storage_id`
9. 返回 `file_table_id + file_record_id` 和必要展示字段

### 为什么文件记录必须保存 `storage_id`

因为文件表当前绑定可以被 `root` 修改，而旧文件不会迁移。

因此：

1. 新上传文件按文件表当前绑定的存储器写入
2. 旧文件继续按自己记录上的 `storage_id` 读取
3. 文件表改绑只影响新上传，不影响旧文件访问

这正是第一版明确要求的行为。

## 读取与下载链路

文件读取时，后端按文件记录本身决定读取来源：

1. 根据文件记录找到 `storage_id`
2. 加载对应存储器实例
3. 通过驱动读取流或生成临时访问 URL

默认建议：

1. 本地和 `rustFS` 都按“默认私有访问”设计
2. 预览和下载优先通过 API 代理或临时 URL 输出
3. `url` 字段只作为展示辅助，不默认视为永久公网 URL

## 文件表改绑规则

第一版规则固定为：

1. `root` 可以把文件表从一个存储器改绑到另一个存储器
2. 改绑只影响之后的新上传
3. 已有文件记录保留原 `storage_id`
4. 不触发历史文件迁移
5. 不补写旧记录的 `url/path/storage_id`

后续如果需要迁移，单独设计“迁移任务”而不是在本轮提前实现。

## 业务引用模型

### 文件表字段引用

如果某个业务模型上的字段本身就绑定某张文件表，那么字段值只需要保存：

1. `file_record_id`

因为字段配置天然知道目标文件表是哪一张。

### 平台内部通用引用

对于聊天附件、图片生成结果等平台内部模块，建议统一保存：

1. `file_table_id`
2. `file_record_id`

这样平台内部模块不需要假设所有文件都来自默认 `attachments`。

## 接口方向

### 存储器管理接口

系统级接口，仅 `root` 可用：

1. 新建存储器
2. 更新存储器
3. 启用 / 禁用存储器
4. 健康检查
5. 设为默认存储器
6. 删除未被引用的存储器

### 文件表管理接口

分两类：

1. `root/system`
   - 查看全部文件表
   - 修改文件表绑定存储器
   - 查看文件表与存储器关系
2. `workspace`
   - 创建自己的文件表
   - 查看自己可见的文件表
   - 删除自己的空文件表或未被使用文件表

### 上传接口

建议保留统一上传入口，但显式带 `file_table_id`：

1. `POST /api/console/files/upload`

输入至少包含：

1. `file_table_id`
2. `multipart file`

如果后续某个页面天然绑定某张文件表，可以在业务 API 内部再包装更窄的上传入口。

## 与现有动态建表能力的关系

本设计不另起一套“文件记录 CRUD 平台”，而是直接复用现有能力：

1. 文件表创建复用 `model_definition` 链路
2. 文件记录 CRUD 复用运行态数据引擎
3. 文件记录权限复用现有 `create/view/edit/delete + own/all`

新增的是：

1. 存储器系统资源
2. 文件表注册系统资源
3. 上传编排与驱动选择

这样可以避免把“文件平台”做成与动态建表体系平行的第二套系统。

## 实现边界建议

第一版建议按以下顺序实现：

1. `storage-object` 抽象补齐
2. 本地驱动落地
3. `rustFS` 驱动落地
4. `file_storages` 与 `file_tables` 控制面表和服务
5. 内建默认 `attachments` 文件表自动创建
6. `workspace` 文件表模板化创建
7. 统一上传接口
8. 下载 / 预览接口
9. 设置页和文件表管理页

## 测试要求

至少覆盖四层：

1. `storage-object` driver 合同测试
   - 本地驱动
   - `rustFS` 驱动
2. 控制面 service 测试
   - 存储器管理
   - 文件表创建
   - 文件表改绑
3. route 测试
   - 上传
   - 下载 / 预览
   - 权限拒绝
4. 运行态回归测试
   - 文件记录按 `own/all` 权限过滤
   - 改绑后旧文件仍可读、新文件走新存储器

## 风险与约束

### 风险一：把文件表和存储器绑定做成强耦合

如果文件记录不保存自己的 `storage_id`，后续一旦改绑，旧文件会立即失联。

结论：

1. 文件记录必须保存实际 `storage_id`

### 风险二：过早开放文件表自定义字段

这会让第一版模板、上传逻辑、预览逻辑和后台管理复杂度暴涨。

结论：

1. 第一版不开放自定义扩展字段

### 风险三：把存储器权限下放到 `workspace`

这会让基础设施边界失控，也与当前 `system/workspace` 分层相冲突。

结论：

1. 存储器和绑定关系都由 `root` 管

## 最终建议

按以下稳定边界落地：

1. `root/system` 管 `file_storages`
2. 系统内建默认真实文件表 `attachments`
3. `workspace` 可创建自己的真实文件表，但只能复用固定模板
4. 文件表当前绑定存储器只影响新上传
5. 文件记录永久保存实际 `storage_id`
6. 文件记录 CRUD 直接复用现有动态建表运行态权限
7. 官方首批只支持 `local` 与 `rustfs`

这样可以在不破坏当前主仓边界的前提下，为后续多模态上传、图片生成产物和多存储扩展留下稳定底座。
