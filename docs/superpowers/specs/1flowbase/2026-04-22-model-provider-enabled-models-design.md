# 1flowbase Model Provider Enabled Models Design

## 背景

截至 `2026-04-22`，模型供应商实例仍按“单个校验模型”建模：

- 实例只保存一个 `validation_model_id`
- 创建 / 编辑抽屉里的“模型检测”同时承担“拉取候选模型”和“选一个校验模型”两种职责
- 运行时消费者默认把实例当成“连接信息 + 单个校验模型”

这套设计已经和当前产品目标不一致。用户明确要求的真实行为是：

- 插件安装到 workspace 后，可以创建多个模型供应商实例
- 每个实例可以主动拉取一次候选模型并缓存下来，后续手动刷新
- 用户可以从候选缓存里快速挑选模型 ID
- 用户也可以手动输入缓存里没有的模型 ID
- 实例真正对外暴露的，不是候选缓存，而是一组“生效模型 ID”
- 后续所有业务在使用这个实例时，只能看到这组“生效模型 ID”

换句话说，候选缓存只是输入辅助池，真正的产品真值是实例上的生效模型数组。

## 目标

本设计要同时满足六个目标：

1. 每个模型供应商实例保存 `enabled_model_ids` 数组，作为实例对外暴露的唯一模型真值。
2. 每个模型供应商实例保存候选模型缓存，作为快速选择池。
3. 候选缓存只在用户主动获取 / 刷新时更新，不自动后台拉取。
4. 用户可以从候选缓存中搜索并添加生效模型，也可以手动输入任意模型 ID 加入生效模型。
5. 刷新候选缓存不会删除已有生效模型，也不会给生效模型增加额外状态标记。
6. “验证实例”语义收口为“获取 / 刷新候选模型缓存”，不再做单模型校验。

## 非目标

本轮明确不做以下事情：

- 不兼容旧的 `validation_model_id` 产品语义。
- 不为旧实例设计自动迁移、兼容读取或回退逻辑。
- 不对“生效模型是否真实存在于上游”做保存时强校验。
- 不给手动输入的生效模型增加“手动模型”“缓存外模型”等额外状态。
- 不让运行时或前端消费者直接消费候选缓存全量结果。

用户已经明确表示当前功能仍在开发中，现有实例可以直接在数据库中删除后按新结构重建，因此本轮按新模型直接收口。

## 方案选择

本次有三种主要方向：

### 方案 A：保留 `validation_model_id`，新增 `enabled_model_ids`

做法：

- 继续保留旧的单值字段用于预览 / 校验
- 新增数组字段用于真正对外可用模型

优点：

- 能少改一部分旧的预览流程

缺点：

- 语义双轨，后续很容易继续混淆
- 前端会长期背负“校验模型”和“生效模型”两套心智

### 方案 B：把 `validation_model_id` 直接升级成 `enabled_model_ids`

做法：

- 删除单值字段
- 所有实例、接口、运行时消费者统一围绕生效模型数组工作

优点：

- 语义最干净
- 后续所有调用点都只认一套真值

缺点：

- 改动面更大，需要同步修改前端、接口、存储和运行时消费

### 方案 C：实例不保存生效模型，只在使用场景临时指定

做法：

- 实例只保存连接信息和候选缓存
- 各业务场景再单独维护允许使用的模型列表

优点：

- 理论上更灵活

缺点：

- 与当前产品心智相反
- 会让同一个实例在不同位置表现不一致

### 结论

采用方案 B。

理由：

- 它和用户确认后的产品心智完全一致：实例本身维护一组生效模型
- 它能彻底去掉“校验模型”这条已经跑偏的旧语义
- 它把后续运行时消费、下拉展示和权限边界都收口到同一份真值

## 数据设计

### 1. 实例真值

模型供应商实例新增并固定保存：

- `enabled_model_ids: string[]`

规则固定为：

- 数组顺序按用户维护顺序保存
- 保存前做去重和空字符串清理
- 允许为空数组

语义固定为：

- 这是实例真正对外暴露给下游选择器和运行时的模型列表
- 下游不得再直接读取候选缓存替代它

### 2. 候选模型缓存

继续复用当前“实例级模型缓存”能力，语义固定为：

- 以实例为粒度保存候选模型列表
- 候选缓存用于创建 / 编辑实例时提供搜索和快速添加
- 不作为运行时下游模型选择真值

规则固定为：

- 新建实例后，如果用户主动点击“获取候选模型”，则将结果缓存到该实例
- 编辑实例时默认直接显示上次缓存结果
- 只有用户主动点击“刷新候选模型”时才重新访问上游并覆盖缓存

## 后端设计

### 1. 领域与端口

`control-plane`、`domain`、`ports` 中关于实例字段的单值模型统一升级为数组模型：

- 删除 `validation_model_id`
- 新增 `enabled_model_ids`

创建 / 更新命令改为显式接收：

- `enabled_model_ids: Vec<String>`
- `preview_token: Option<Uuid>`

这里的 `preview_token` 不再表示“我已经选好了哪个校验模型”，而只表示：

- 本次保存可以复用刚刚拉取到的候选模型结果

如果没有 `preview_token`：

- 允许仅保存实例配置和 `enabled_model_ids`
- 不强制拉取候选缓存

### 2. 候选缓存刷新

原“验证实例”入口改造成“获取 / 刷新候选模型缓存”：

- 调用 provider 侧模型发现能力
- 将返回模型保存为该实例的候选缓存
- 返回候选缓存结果给前端展示

不再做以下动作：

- 不再要求传入单个校验模型
- 不再对某个具体模型做额外验证
- 不再根据单个模型去推导实例状态

### 3. 保存逻辑

创建 / 编辑实例的保存逻辑固定为：

1. 规范化 `enabled_model_ids`
2. 保存实例配置
3. 写入 `enabled_model_ids`
4. 若本次携带 `preview_token`，则复用该 token 对应的候选模型结果写入实例缓存
5. 删除本次 preview session

其中：

- `enabled_model_ids` 不要求必须存在于候选缓存中
- 刷新候选缓存时，不得清理已有 `enabled_model_ids`

### 4. 运行时消费

所有运行时和下游选择器消费者统一改成只读取实例的 `enabled_model_ids`。

产品规则固定为：

- 候选缓存不再直接暴露给业务选择器
- 如果实例没有任何 `enabled_model_ids`，则该实例对下游相当于“无可用模型”

## 存储设计

数据库层直接切换到新结构，不做兼容保留：

- 在模型供应商实例表中删除 `validation_model_id`
- 新增 `enabled_model_ids` 数组列

推荐数据库表达：

- PostgreSQL `text[]`

仓储层责任固定为：

- 原样读写 `enabled_model_ids`
- 继续维护实例级候选模型缓存表

本轮不引入单独的“enabled models 子表”，因为当前只需要保存一组有序字符串值，数组列已经足够表达。

## 前端交互设计

### 1. 抽屉结构

创建 / 编辑实例抽屉中的“模型检测”区域改名并拆成两块：

- `候选模型`
- `生效模型`

其中：

- `候选模型` 用于展示当前缓存、触发获取 / 刷新
- `生效模型` 用于维护最终保存的模型 ID 数组

### 2. 候选模型区

候选模型区固定提供：

- “获取候选模型”按钮
- 已缓存的候选模型列表或空态说明

行为固定为：

- 新建实例时默认无缓存，用户主动点击后才拉取
- 编辑实例时优先显示实例已有缓存
- 再次点击按钮时覆盖更新缓存

### 3. 生效模型区

生效模型区使用可搜索、可多选、可手动输入的标签式输入控件。

能力固定为：

- 从候选缓存中搜索并选择多个模型 ID
- 允许直接输入任意模型 ID 并回车加入
- 支持删除任意已加入的模型 ID
- 不展示“手动添加”或“缓存内”之类的状态标签

保存时固定只提交：

- `enabled_model_ids`
- 可选 `preview_token`

不再提交：

- `validation_model_id`

### 4. 实例列表与详情

实例列表和实例详情中的展示口径统一改为：

- 展示生效模型数量
- 展示部分已生效模型 ID
- 若需要展开查看，展开区只围绕“生效模型”和“候选缓存”组织，不再出现“校验模型”文案

## API 合同设计

### 创建 / 更新实例

请求体改为接收：

- `display_name`
- `config`
- `enabled_model_ids`
- `preview_token`

响应体中的实例字段改为返回：

- `enabled_model_ids`

删除：

- `validation_model_id`

### 获取 / 刷新候选模型

保留现有预览 / 拉取接口的大体职责，但对外语义改名为“候选模型”：

- 返回当前拉取到的候选模型列表
- 返回 `preview_token`

前端在用户后续保存时可携带该 `preview_token`，避免重复写缓存。

## 错误处理

保存实例时：

- `enabled_model_ids` 非法类型直接返回 `invalid input`
- 数组内空字符串在服务端清理后保存，不单独报错
- 重复模型 ID 自动去重，不作为错误

刷新候选缓存时：

- 上游拉取失败，只影响候选缓存刷新结果，不修改已保存的 `enabled_model_ids`

## 测试设计

本轮至少覆盖以下回归：

### 后端

- 创建实例时可以保存空的 `enabled_model_ids`
- 创建实例时可以保存多个 `enabled_model_ids`
- 创建实例时允许 `enabled_model_ids` 包含候选缓存里没有的值
- 携带 `preview_token` 保存时会复用候选缓存
- 刷新候选缓存不会删除已有 `enabled_model_ids`
- 运行时消费者只读取 `enabled_model_ids`

### 前端

- 抽屉里可以从候选缓存中搜索并加入多个模型
- 抽屉里可以手动输入模型 ID 并加入
- 保存请求发送 `enabled_model_ids`
- 编辑实例时可以显示已有 `enabled_model_ids`
- 刷新候选缓存不会清空已编辑的生效模型输入值

## 实施顺序

建议按以下顺序实现：

1. 后端领域模型、接口 DTO、数据库迁移、仓储读写统一替换为 `enabled_model_ids`
2. 候选缓存 / preview session 逻辑从“单模型校验复用”收束为“候选缓存复用”
3. 前端抽屉、实例列表、实例详情改成“候选缓存 + 生效模型数组”交互
4. 运行时消费者和下游选择器切到 `enabled_model_ids`
5. 完成定向回归测试

## 验收标准

当以下条件全部成立时，本设计视为落地完成：

1. 新建实例时，不必选择单个校验模型，允许直接维护多个生效模型。
2. 用户可以从候选缓存中快速添加模型，也可以手动输入任意模型 ID。
3. 刷新候选缓存不会影响已保存的生效模型。
4. 业务实际使用实例时，下拉里只出现实例的 `enabled_model_ids`。
5. 前后端和数据库中不再残留 `validation_model_id` 作为产品真值。
