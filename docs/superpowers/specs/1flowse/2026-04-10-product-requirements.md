# 1Flowse P1 需求文档

日期：2026-04-10
状态：已完成初稿，待用户审阅
关联设计稿：[2026-04-10-product-design.md](./2026-04-10-product-design.md)

## 1. 文档目标

本文档用于把已经确认的产品设计，收敛为可执行的 P1 产品需求范围。

本文档重点回答：
- P1 必须提供哪些用户能力
- 哪些能力是核心闭环的一部分
- 哪些能力暂时不做
- 后续架构文档需要对齐哪些约束

## 2. P1 产品目标

P1 的核心目标是让一个团队能够完成以下闭环：

1. 创建一个 AI 应用。
2. 通过画布编排一个或多个 Flow。
3. 在 Flow 中组合多个 LLM 节点、工具节点和状态读写节点。
4. 将 Flow 发布成标准 Agent 友好的 API。
5. 让外部调用方稳定调用该应用。
6. 在运行过程中支持工具调用暂停、回调恢复和跨会话状态延续。
7. 通过日志、监控和权限控制对应用运行进行管理。

## 3. 目标用户与使用场景

### 3.1 目标用户

- `团队管理员`
  - 负责团队成员、角色、资源可见性和后台配置
- `应用构建者`
  - 负责创建应用、编排 Flow、配置状态模型、发布接口
- `应用维护者`
  - 负责查看日志、分析监控、排查失败调用、调整配置
- `外部调用方`
  - 通过标准 API 调用应用，可能是 Codex、Claude Code、本地 Agent 或其他客户端

### 3.2 核心使用场景

- 团队内部搭建 AI 工作流并作为标准能力发布
- 使用状态表保存跨会话记忆、用户状态或业务上下文
- 在日志中查看单次会话内的节点执行轨迹和工具调用链路
- 通过监控判断 token 成本、请求量、失败率和热点节点
- 通过权限策略控制谁可以查看、修改、发布或使用应用资源

## 4. 范围定义

### 4.1 P1 包含

- 团队空间模型
- 应用管理
- Flow 画布编排
- 多 LLM 节点编排
- 工具调用暂停与恢复
- callback 回填恢复
- checkpoint 持久化
- 统一内部协议与标准 Agent 适配发布
- 状态模型与基础 CRUD
- 外部数据源接入与使用控制
- 日志与监控页面
- 团队角色与资源权限
- 基础后台管理页面
- 内置供应商插件和节点插件

### 4.2 P1 不包含

- 个人空间
- 完整多租户治理
- 复杂低代码页面搭建器
- 完整插件市场
- 通用低代码平台能力
- 以平台内聊天为中心的产品模式

## 5. 功能需求

### 5.1 工作台与应用管理

### FR-001 应用列表

系统必须提供 `工作台` 页面，用于展示团队空间下的应用列表。

### FR-002 应用创建

用户必须可以在工作台中创建应用，并为应用设置名称、描述和基础配置。

### FR-003 应用检索

工作台必须支持应用搜索、筛选和最近访问能力。

### FR-004 应用概览

工作台必须展示应用的基础状态信息，至少包括：
- 发布状态
- 最近调用时间
- 近期调用量概览

### 5.2 团队与权限

### FR-010 团队成员管理

系统必须提供 `团队` 页面，用于管理成员加入、移除和成员基础信息。

### FR-011 角色管理

系统必须支持轻量角色模型，用于表达不同成员对资源的访问能力。

### FR-012 资源授权

系统必须支持对以下资源进行授权：
- Flow
- Publish Endpoint
- State Model
- State Data
- External Data Source
- Plugin Config
- Admin Route / View

### FR-013 授权动作

系统必须至少支持以下动作级权限：
- `view`
- `edit`
- `publish`
- `manage`
- `use`
- `configure`

### FR-014 可见性策略

系统必须支持以下基础策略：
- 仅创建者可见
- 团队成员可见
- 团队成员可修改
- 指定角色可管理
- 指定角色可发布或调用

### FR-015 路由权限联动

后台路由访问必须与资源授权联动，用户不能仅因拥有页面入口而绕过资源权限。

### 5.3 画布编排

### FR-020 无限画布

系统必须提供基于 `xyflow` 的无限画布，用于编排节点与连线。

### FR-021 Flow 编辑

用户必须可以创建、编辑、复制和版本化管理 Flow。

### FR-022 节点配置

用户必须可以配置节点输入、输出、变量绑定、运行参数和节点级元数据。

### FR-023 多节点类型

P1 必须至少支持以下节点类别：
- LLM 节点
- 工具节点
- 状态读取节点
- 状态写入节点
- 输入输出节点

### FR-024 多 LLM 节点

单个 Flow 必须支持多个 LLM 节点参与执行，并允许节点间传递统一内部对象。

### FR-025 发布配置

Flow 编辑过程中，用户必须可以配置发布相关信息，包括接口行为、鉴权策略和输入输出契约。

### 5.4 运行时与执行状态

### FR-030 Flow Run

系统必须记录每次 Flow 调用的运行实例，并保留可查询的运行上下文。

### FR-031 Node Run

系统必须记录节点级执行实例，以支持节点输入输出、暂停恢复和错误排查。

### FR-032 Flow Run 状态

系统必须支持以下 Flow Run 状态：
- `queued`
- `running`
- `waiting_callback`
- `waiting_human`
- `paused`
- `succeeded`
- `failed`
- `cancelled`

### FR-033 Node Run 状态

系统必须支持以下 Node Run 状态：
- `pending`
- `ready`
- `running`
- `streaming`
- `waiting_tool`
- `waiting_callback`
- `waiting_human`
- `retrying`
- `succeeded`
- `failed`
- `skipped`

### FR-034 工具调用暂停

当 LLM 节点触发工具调用时，系统必须暂停当前节点执行，而不是直接推进到下游节点。

### FR-035 checkpoint 持久化

当运行进入暂停或等待回调阶段时，系统必须持久化 checkpoint，以支持恢复执行。

### FR-036 callback 恢复

当外部 callback 数据回填后，系统必须先恢复当前暂停节点，再推进后续节点执行。

### FR-037 流式输出

系统应支持运行时流式输出能力，以适配主流 Agent 交互方式。

### 5.5 发布与 API 文档

### FR-040 统一内部协议

P1 必须先定义一套统一内部运行协议，作为所有对外发布能力的中间层。

### FR-041 对外适配层

系统必须允许在统一内部协议之上，为外部 Agent 场景提供适配层。

### FR-042 发布接口

用户必须可以把 Flow 发布为标准 Agent 友好的接口，并获得稳定访问入口。

### FR-043 API 文档页

每个应用必须提供 `API 文档` 页面，至少展示：
- 接口形态
- 鉴权方式
- 请求格式
- 响应格式
- callback 方式
- 示例调用

### FR-044 AI 友好 API

对外 API 设计必须尽量减少冗长参数和不必要嵌套，以适配 AI 生成调用参数时的约束。

### 5.6 状态与数据

### FR-050 状态模型管理

系统必须允许用户在应用内定义状态模型，用于保存记忆和业务状态。

### FR-051 关系定义

状态模型必须支持基础关系表达，以满足跨对象状态管理需求。

### FR-052 CRUD 生成

状态模型创建后，系统必须自动生成基础 CRUD 能力。

### FR-053 数据浏览与管理

系统必须提供 `状态与数据` 页面，用于浏览、查询和修改状态数据。

### FR-054 数据源接入

系统必须支持接入外部数据源，并允许 Flow 或状态模型按权限使用这些数据源。

### FR-055 数据源权限

系统必须区分：
- 谁可以创建或配置数据源
- 谁可以查看数据源元信息
- 谁可以使用数据源
- 谁可以管理凭据

### FR-056 上下文注入

状态模型中的数据必须可以按规则注入 Flow 运行上下文，作为跨会话记忆或业务上下文来源。

### 5.7 对话日志与监控

### FR-060 对话日志页

每个应用必须提供 `对话日志` 页面，且其定位是运行轨迹页，不只是消息列表。

### FR-061 日志内容

每条运行日志至少应包含：
- 对话内容
- 工具调用记录
- 节点输入输出
- 暂停与恢复事件
- callback 轨迹
- 错误信息
- 最终结果

### FR-062 监控页

每个应用必须提供 `监控` 页面，展示用户最关心的调用与成本指标。

### FR-063 关键监控指标

监控页至少展示：
- 单次对话 token 消耗
- 今日请求次数
- 应用调用总量
- 成功率
- 失败率
- 平均响应时长
- 工具调用次数
- 模型供应商消耗分布
- 高频节点
- 失败高发节点
- 近期异常会话数

### 5.8 插件与供应商扩展

### FR-070 插件框架

系统必须提供统一插件框架，用于承接模型供应商、节点、数据源和发布适配扩展。

### FR-071 内置插件

P1 必须先提供可直接使用的内置供应商插件和节点插件。

### FR-072 节点扩展能力

插件体系必须支持新增节点类型，以便后续扩展 Flow 编排能力。

### FR-073 发布扩展能力

插件体系应允许未来通过适配插件扩展更多对外协议和发布模式。

### 5.9 AI 辅助能力

### FR-080 AI Assistant 侧栏

系统可以提供 AI Assistant 侧栏或上下文助手，但其定位必须是辅助面，而非主产品入口。

### FR-081 AI 辅助范围

AI Assistant 至少可用于辅助：
- 节点配置
- Flow 编辑说明
- 前端代码片段生成
- 状态建模建议
- 错误解释

### FR-082 上下文绑定

AI Assistant 必须绑定当前应用上下文、当前编辑对象和当前运行上下文，不能脱离场景独立工作。

## 6. 非功能需求

### NFR-001 可恢复性

运行时必须在暂停、回调和关键边界具备可恢复能力，避免长链路调用因中断而全部丢失。

### NFR-002 可观测性

日志、监控、节点状态、工具调用和错误信息必须可查询、可追踪。

### NFR-003 API 易用性

API 必须对 AI 和人类调用方都尽量友好，避免不必要的复杂参数结构。

### NFR-004 扩展性

系统必须允许后续以插件方式扩展模型供应商、节点能力、数据源和发布适配。

### NFR-005 权限一致性

资源权限和路由权限必须由后端统一治理，避免出现前端可见但后端不可用，或后端泄漏能力的情况。

### NFR-006 凭据安全

数据源和供应商配置中的敏感凭据必须被安全存储，并与普通资源元信息分离管理。

### NFR-007 成本可见性

系统必须尽可能让用户理解 token、请求量和调用成本，而不是把成本信息隐藏在底层。

## 7. P1 验收标准

P1 验收应至少满足以下条件：

- 团队可以创建并管理应用
- 用户可以在画布中编排多节点 Flow
- Flow 可以发布成标准 Agent 友好的接口
- 外部客户端可以成功调用已发布接口
- 系统可以正确处理工具调用暂停与 callback 恢复
- 状态模型可以支撑跨会话上下文延续
- 日志页面可以追踪单次运行的完整链路
- 监控页面可以展示调用量、成本和稳定性指标
- 权限体系可以控制资源与路由访问
