## 1. 文档目标

本文档用于把 1Flowse 的“数据建模定义”正式收敛到一条可实现、可扩展、可维护的后端路线。

本文档要解决的核心问题是：

- 数据建模定义到底是“元数据声明”，还是“直接驱动 PostgreSQL 物理表”
- 用户修改数据结构后，是否需要发布
- 运行时统一路由如何映射到真实物理表
- 字段显示提示、动态界面和页面布局之间的边界如何划分
- 插件如何在这条链上扩展，但不污染宿主内核

本文档只覆盖当前已确认范围：

- 数据建模定义控制面
- PostgreSQL 物理表生成与同步变更
- 运行时统一 CRUD 路由
- 字段显示提示与动态界面消费边界
- 宿主扩展、运行时扩展、能力插件的挂载边界

本文档不覆盖：

- 复杂报表、聚合分析、数据仓库产品能力
- 字段级精细权限
- 异步迁移作业、灰度发布、大表在线变更
- 文件、通知、任务调度等未进入本轮讨论的业务域

## 2. 设计结论

### 2.1 总体结论

1Flowse 的数据建模定义正式采用：

`控制面保存定义 + 实时变更 PostgreSQL 物理表 + 统一运行时路由访问真实表`

含义如下：

- 数据建模定义不是“发布后才变成真实数据能力”的草稿系统。
- 用户有权限执行结构修改时，系统同步修改 PostgreSQL 物理表并立即生效。
- 对外仍保持统一运行时路由，不直接暴露物理表路径。
- 数据建模定义与界面布局分离。
- 字段元数据可以携带默认显示提示，但页面如何组合与展示由动态界面层决定。

### 2.2 明确不采用的方向

以下方向不作为当前正式路线：

- 不采用“数据建模定义发布后才生成运行时资源”的版本发布流程。
- 不采用“运行时数据全部存进统一通用表或 JSON 容器”的统一存储引擎路线。
- 不采用“先异步受理，后台慢慢改表”的结构变更主流程。
- 不采用“物理表名、物理字段名、物理字段类型可随意修改”的灵活模式。

### 2.3 七条固定规则

1. 结构修改有权限即可实时生效，不做用户可见发布流程。
2. 物理表名在创建时固定，不允许重命名。
3. 物理字段名和物理字段类型在创建时固定，不允许重命名或改类型。
4. 删除表、删除字段、删除关系允许执行，但必须显式确认。
5. 运行时对外继续走统一路由，内部映射到 PostgreSQL 真表。
6. 数据建模定义作用域在创建时固定，只允许 `team` 或 `app` 二选一。
7. 所有结构变更同步执行；失败则回滚，成功则返回成功结果并提示前端刷新数据。

## 3. 核心对象与边界

### 3.1 数据建模定义

数据建模定义负责：

- 数据表编码与标题
- 作用域
- 物理表映射
- 字段定义
- 关系定义
- 索引定义
- 权限与审计基础命名空间

数据建模定义不负责：

- 页面布局
- 区块拖拽结构
- 表单排版
- 列表列顺序
- 详情页布局

### 3.2 字段显示提示

字段元数据可以携带默认显示提示，例如：

- 默认输入组件类型
- 枚举字段的选项配置
- 关系字段的默认展示组件
- 基础只读显示配置
- 基础占位提示、描述、显示标题

边界如下：

- 这些提示只服务于动态界面层的默认渲染。
- 它不是页面布局定义。
- 它不决定某个字段一定出现在什么页面位置。
- 动态界面代码可以消费、忽略或覆盖这些提示。

### 3.3 页面布局定义

页面布局定义负责：

- 哪个页面使用哪些字段
- 字段在列表、表单、详情中的展示顺序
- 哪些字段分组显示
- 关联字段以嵌套、子表或弹窗方式展示
- 页面区块、操作栏、详情标签页等界面结构

因此最终边界固定为：

- 数据建模定义负责“数据是什么”
- 字段显示提示负责“默认建议怎么显示”
- 页面布局定义负责“这个页面具体怎么摆、怎么用”

## 4. 元数据模型

### 4.1 数据建模定义主表

系统保留数据建模定义主表，用于保存控制面元数据。最少字段包括：

- `id`
- `scope_kind`
- `scope_id`
- `code`
- `title`
- `physical_table_name`
- `acl_namespace`
- `audit_namespace`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

约束：

- `code` 在所属作用域内唯一。
- `title` 可修改。
- `physical_table_name` 创建后固定。
- `scope_kind` 创建后固定，只允许 `team` 或 `app`。

### 4.2 字段定义表

系统保留字段定义表，用于保存字段控制面元数据。最少字段包括：

- `id`
- `data_model_id`
- `code`
- `title`
- `physical_column_name`
- `field_kind`
- `is_required`
- `is_unique`
- `default_value`
- `display_interface`
- `display_options`
- `relation_target_model_id`
- `relation_options`
- `sort_order`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

约束：

- `code` 在所属数据模型定义内唯一。
- `physical_column_name` 创建后固定。
- 标量字段和关系字段都通过字段定义承载。

### 4.3 结构变更日志

虽然不采用用户可见发布版本，但仍保留内部结构变更日志，用于：

- 审计谁改了什么
- 回看失败原因
- 排查多实例不同步问题
- 为未来迁移工具保留基础

日志至少记录：

- 变更动作
- 目标对象
- 操作人
- 变更前摘要
- 变更后摘要
- 执行结果
- 错误信息
- 时间

## 5. 物理表与物理字段规则

### 5.1 表标识规则

每个数据模型定义同时有三类名称：

- `code`
  - 机器编码，面向运行时路由和控制面识别
- `title`
  - 展示名称，面向用户界面
- `physical_table_name`
  - PostgreSQL 物理表名，创建后固定

推荐规则：

- `code` 使用小写短横线或下划线风格
- `title` 允许自由修改
- `physical_table_name` 由宿主在创建时生成，例如 `rtm_<scope-kind>_<short-key>_<code>`

生成后：

- `title` 可以改
- `code` 创建后固定
- `physical_table_name` 永远不允许改

### 5.2 自动注入系统字段

每张运行时物理表自动注入以下字段：

- `id`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

同时按作用域自动注入以下字段之一：

- `team_id`
- `app_id`

规则如下：

- `team` 作用域表必须带 `team_id`
- `app` 作用域表必须带 `app_id`
- 作用域字段由宿主生成和维护，不允许用户删除或改类型

### 5.3 标量字段

第一阶段支持的标量字段范围固定为：

- 字符串
- 数字
- 布尔
- 时间
- 枚举
- 文本
- JSON

规则如下：

- 字段编码创建后固定
- 物理字段类型创建后固定
- 唯一索引允许开启和关闭
- 不支持字段改名
- 不支持字段改类型

如需改名或改类型，正式处理方式固定为：

- 删除旧字段
- 新建新字段
- 如需迁移旧数据，后续由专门工具处理，不进入当前主流程

### 5.4 一对多 / 多对一

关系落地规则固定为：

- `多对一` 是拥有物理外键的一侧
- `一对多` 是读侧或反向关系视图，不额外创建新列

也就是说：

- 真实 PostgreSQL 外键列由 `多对一` 字段生成
- `一对多` 只反映关系能力和运行时展开方式

### 5.5 多对多

多对多关系统一由宿主生成中间表。

第一阶段约束：

- 中间表为宿主托管
- 中间表名由宿主生成并固定
- 中间表至少包含两侧外键和系统审计字段
- 不开放自定义复杂中间模型

## 6. 同步结构变更流程

### 6.1 创建数据模型定义

创建流程固定为：

1. 校验作用域、编码、权限
2. 生成固定 `physical_table_name`
3. 写入数据建模定义元数据
4. 在 PostgreSQL 中创建物理表并注入系统字段
5. 从 PostgreSQL 元数据重建该数据模型定义的运行时注册信息
6. 返回成功结果，并提示前端刷新数据

### 6.2 新增字段

新增字段流程固定为：

1. 校验字段编码、类型、权限
2. 写入字段元数据
3. 执行 PostgreSQL `ALTER TABLE ADD COLUMN`
4. 如有唯一索引或外键，继续补齐约束与索引
5. 增量刷新运行时注册信息
6. 返回成功结果

### 6.3 删除字段

删除字段流程固定为：

1. 前端显式确认
2. 校验权限与宿主保护规则
3. 删除字段元数据
4. 执行 PostgreSQL 删列或删关系约束
5. 刷新运行时注册信息
6. 返回成功结果

### 6.4 删除数据模型定义

删除流程固定为：

1. 前端显式确认
2. 校验权限
3. 删除相关字段元数据与运行时注册记录
4. 删除 PostgreSQL 物理表及宿主生成的关系中间表
5. 返回成功结果

### 6.5 事务与回滚

当前主流程固定为同步事务式执行：

- 元数据写入与 PostgreSQL 结构变更必须属于同一轮同步操作
- 任一步失败，当前请求直接返回失败
- PostgreSQL 可事务化的结构变更必须回滚
- 只有数据库状态和运行时注册状态都完成刷新后，才返回成功

不采用：

- 后台异步迁移
- 队列化发布
- 用户提交后稍后生效

## 7. 运行时统一路由

### 7.1 路由口径

运行时继续保留统一路由口径，例如：

- `GET /api/runtime/models/:model_code/records`
- `POST /api/runtime/models/:model_code/records`
- `GET /api/runtime/models/:model_code/records/:record_id`
- `PATCH /api/runtime/models/:model_code/records/:record_id`
- `DELETE /api/runtime/models/:model_code/records/:record_id`

关系与动作可继续沿统一风格扩展，但当前主线先固定基础 CRUD。

### 7.2 内部解析方式

统一路由收到请求后，宿主必须解析出：

- 数据模型定义编码
- 对应物理表名
- 作用域字段
- 可访问字段集合
- 关系展开规则
- 查询范围限制
- 审计命名空间

然后再将请求代理到 PostgreSQL 仓储实现。

### 7.3 第一阶段查询能力

第一阶段固定支持：

- 列表
- 详情
- 新增
- 修改
- 删除
- 基础筛选
- 排序
- 分页
- 按关系展开

不在第一阶段支持：

- 复杂聚合
- 跨模型任意联表查询构造器
- 公式字段执行引擎
- 物化视图

## 8. 权限与作用域

### 8.1 数据建模定义级权限

宿主必须先控制：

- 谁能创建数据模型定义
- 谁能删数据模型定义
- 谁能加字段、删字段、改显示标题
- 谁能改唯一索引与关系

### 8.2 记录范围级权限

运行时记录访问第一阶段只做到：

- 数据模型定义级权限
- 记录范围级权限

不做：

- 字段级精细权限

### 8.3 固定作用域

数据建模定义创建时必须选择：

- `team`
- `app`

二选一后不可切换。

其后果是：

- 物理表作用域字段固定
- 记录范围控制入口固定
- 插件绑定时的查询范围入口固定

## 9. 字段显示提示与动态界面

### 9.1 字段元数据可携带的显示提示

字段元数据第一阶段允许携带：

- 默认显示组件类型
- 枚举显示配置
- 关系字段默认组件类型
- 基础标题、描述、占位提示
- 基础只读显示方式

### 9.2 动态界面层职责

动态界面层负责：

- 读取字段显示提示
- 决定默认表单组件
- 决定默认列表列组件
- 决定默认详情渲染
- 在不同页面场景下覆盖默认显示提示

后端不负责页面布局拼装，只负责提供字段显示提示和字段元数据。

## 10. 插件挂载边界

### 10.1 宿主扩展

`host-extension` 可以扩展：

- 字段类型注册
- 结构变更钩子
- 运行时资源注册
- 宿主级数据建模定义能力

### 10.2 运行时扩展

`runtime extension` 只能绑定到宿主预定义槽位，例如：

- 默认值
- 校验
- 查询范围
- 记录动作

它不能：

- 注册新的外部 HTTP 接口
- 直接改 PostgreSQL 结构
- 绕过宿主资源注册中心

### 10.3 能力插件

`capability plugin` 只负责执行能力，例如：

- 节点执行
- 外部读取
- 发布渲染
- 发布投递

它不直接参与数据建模定义结构变更。

## 11. 多实例与注册中心

### 11.1 单一事实源

PostgreSQL 中的数据建模定义元数据是唯一事实源。

内存中的运行时资源注册表只是缓存和加速层，不是事实源。

### 11.2 启动与增量刷新

固定策略如下：

- 服务启动时，从 PostgreSQL 全量重建运行时资源注册表
- 单次结构修改成功后，对受影响的数据模型定义做增量刷新

### 11.3 多实例基本策略

第一阶段至少保证：

- 任一实例重启后可从 PostgreSQL 重新构建完整注册表
- 不允许只依赖单节点内存状态长期运行

后续如进入多实例正式生产，再补跨节点失效通知或注册表刷新广播机制。

## 12. 与当前 1Flowse 代码的对齐要求

当前代码里与本设计冲突的旧口径主要有两类：

- “数据模型定义发布后再形成运行时数据资源”
- “运行时数据先落内存引擎再逐步替换”

后续实现必须改到以下新口径：

- 去掉用户可见发布流程
- 去掉把运行时数据建立在内存引擎上的主路径
- 让数据建模定义直接驱动 PostgreSQL 真表
- 让运行时统一路由直接访问 PostgreSQL 仓储

## 13. 非目标

以下内容明确不进入本轮：

- 字段重命名工具
- 字段类型迁移工具
- 大表在线变更优化
- 异步结构变更队列
- 字段级权限
- 复杂分析查询构建器
- 用户自定义中间模型

## 14. 最终结论

1Flowse 的数据建模定义正式采用 NocoBase 类实时物理表路线：

- 控制面保存元数据
- 修改后同步变更 PostgreSQL 物理表
- 不做用户可见发布流程
- 运行时统一路由访问真实表
- 字段只提供默认显示提示
- 动态界面代码决定如何渲染字段
- 页面布局继续独立管理
- 插件继续沿宿主扩展、运行时扩展、能力插件三层边界工作

这条路线的目标不是追求“最灵活的元数据实验系统”，而是优先得到：

- 对用户直觉成立的数据建模体验
- 对 PostgreSQL 真实可见、可分析的物理表结构
- 对插件边界和运行时数据都可长期维护的宿主内核
