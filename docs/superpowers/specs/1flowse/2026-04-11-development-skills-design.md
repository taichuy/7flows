# 1Flowse 前后端开发 Skill 设计稿

日期：2026-04-11
状态：待用户审阅

## 1. 文档目标

本文档用于定义两类开发 Skill 的设计方向：

- `frontend-development`
- `backend-development`

目标不是复述项目规格，而是沉淀一套稳定、可复用、可迁移的开发约束，帮助 AI 在开发过程中减少代码腐化、控制变化传播、避免项目逐步演化为难维护的“屎山”。

## 2. 核心设计结论

本次 Skill 设计采用：

- `原则型 + 检查清单 + 少量示例`

不采用：

- `项目 spec 搬运`
- `技术栈手册`
- `目录结构镜像`
- `过多场景化分支`

原因如下：

- 项目文档会过时，Skill 不应依赖大量易过期事实。
- Skill 应提供稳定的工程判断规则，而不是复述某个项目当前状态。
- 真正决定代码是否容易腐化的，通常不是框架名字，而是边界、职责、状态、一致性和变化传播控制。

## 3. Skill 范围与非目标

### 3.1 范围

Skill 只负责回答以下问题：

- 什么时候必须先问人，什么时候可以直接做。
- 怎么判断该不该拆文件、拆模块、拆接口。
- 怎么避免变化传播。
- 怎么保证状态、一致性、唯一入口。
- 怎么判断 API 是否对 AI 友好。
- 哪些实现方式属于反模式。
- 开发完成后要自查什么。

### 3.2 非目标

Skill 不负责承载以下内容：

- 项目当前技术栈事实
- 具体模块边界定稿结果
- 项目专有路径与目录约定
- 当前系统的固定状态名、接口名、路由名
- 对某个项目 spec 的全文摘要

这些内容应继续保留在项目文档、根规则或用户记忆中。

## 4. Skill 组织方式

建议采用如下结构：

```text
.agents/skills/
  frontend-development/
    SKILL.md
    references/
      communication-gate.md
      review-checklist.md
      anti-patterns.md
      examples.md
  backend-development/
    SKILL.md
    references/
      api-design.md
      state-and-consistency.md
      boundary-design.md
      anti-patterns.md
      examples.md
```

组织原则：

- `SKILL.md` 只保留触发条件、铁律、核心原则、执行流程、引用关系。
- `references/` 只放重内容：清单、反模式、示例。
- 示例保持少量且通用，不绑定单个项目事实。

## 5. 共通设计原则

两个 Skill 应共享以下总原则：

1. 以“防止变化传播”为最高约束之一，而不是只追求当前改动能跑通。
2. 单一职责按“变化原因”判断，而不是按函数数量或文件长度做机械切分。
3. 优先复用稳定契约和成熟实现，不为“看起来模块化”做无意义抽象。
4. 外部变化不应直接污染核心规则；内部实现允许演化，但对外契约应保持稳定。
5. 任何会改动产品体验、核心状态、对外契约、系统边界的事项，都应优先确认人类决策。
6. Skill 不是项目真相来源，只是开发过程中的判断器与防腐层。

## 6. Frontend Skill 设计

### 6.1 目标

`frontend-development` 用于约束前端代码与界面实现，重点解决以下问题：

- 页面与组件越堆越乱
- 交互规则不一致
- 状态散落、难以追踪
- 为了抽象而抽象
- 为了赶进度临时拼接出不可维护结构

### 6.2 必须人工确认的场景

以下场景应先与人沟通，再进入实现：

- 新增页面
- 新增流程
- 调整交互流
- 调整视觉方案
- 增加页面中的 AI 协作层能力

如果环境中存在专门处理信息架构和交互一致性的 Skill，例如 `frontend-logic-design`，则在以下场景必须联动：

- 信息架构设计
- 交互一致性审查
- 页面层级重组
- 导航与入口调整

### 6.3 可以直接按规则执行的场景

以下场景可直接按 Skill 执行：

- 在既有边界内补实现
- 在既有视觉语法内补局部交互
- 复用已有组件完成同类页面
- 修复明显的职责混杂、状态散落、重复拼装问题

### 6.4 核心原则

`frontend-development` 应至少包含以下原则：

1. 组件、页面、协议、嵌入层边界清楚，避免一个前端文件同时承载展示、状态编排、远程协议拼接和页面路由职责。
2. 优先复用现有组件与成熟依赖，不为单点使用的代码强行抽组件。
3. 页面状态、异步状态、表单状态、运行态展示状态应尽量分层管理，避免全部堆到页面根组件。
4. 不随手创造新的视觉语法、交互手势或命名体系；新增实现应尽量服从已有系统。
5. 是否拆文件、拆组件，应优先依据变化原因和复用价值判断，而不是只依据行数。
6. 页面中一旦出现“同类对象行为不一致”，应视为结构性问题，而不是局部小瑕疵。

### 6.5 检查清单

应至少包含三段检查：

#### 开发前

- 这是新增页面/流程/交互/视觉方案吗？
- 是否已有现成组件、模式或成熟依赖可复用？
- 这次改动会不会引入新的视觉语法或新的交互规则？
- 状态该放在页面、组件、共享层还是协议层？

#### 开发中

- 当前文件是否开始承担多个变化原因？
- 是否出现“为了复用而复用”的提前抽象？
- 是否把多个不同类型状态堆在一起？
- 是否把协议细节、UI 展示细节和业务判断写进同一个位置？

#### 开发后

- 同类组件行为是否一致？
- 交互是否仍符合既有页面语法？
- 是否引入新的隐藏状态或难以追踪的局部逻辑？
- 这次改动后，后续需求是否必须继续改同一大片代码？

### 6.6 反模式

应明确列出的前端反模式至少包括：

- 一个页面组件同时管理路由、请求编排、复杂表单、弹窗状态和视觉布局
- 同类组件采用不同交互行为
- 为单次使用抽出过早组件或 hooks
- 随意新增新的按钮样式、弹窗语义或操作路径
- 页面内把协议拼装、数据转换和展示层混写

### 6.7 示例要求

示例只保留少量、通用、跨项目可迁移的内容，例如：

- 什么时候不该抽公共组件
- 什么时候页面状态应提升到共享层
- 什么时候界面问题应交给专门的信息架构 Skill

## 7. Backend Skill 设计

### 7.1 目标

`backend-development` 用于约束后端接口、模块边界、状态设计与一致性处理，重点解决以下问题：

- 业务规则和外部协议耦合在一起
- 接口越来越长、越来越深、越来越难让 AI 正确构造
- 状态流转缺少唯一入口
- 多个模块都能直接修改核心状态
- 一次小改动引发连锁修改

### 7.2 必须人工确认的场景

以下场景应先与人沟通，再进入实现：

- 改动核心状态机
- 改动对外发布协议
- 改动权限策略
- 改动插件边界
- 改动核心对象定义

### 7.3 可以直接按规则执行的场景

以下场景可直接按 Skill 执行：

- 在既有核心模型下补充实现
- 对适配层做局部收敛和隔离
- 将隐式状态流转改为显式入口
- 将过长、过深、职责混杂的接口收敛为更清晰的单一动作

### 7.4 核心原则

`backend-development` 应至少包含以下原则：

1. 先区分稳定核心与边界适配层；核心承载业务规则，适配层处理外部协议、依赖和实现细节。
2. 业务层决定“该不该做”，适配层处理“怎么做到”，禁止把外部协议细节直接写进核心业务规则。
3. 对外契约保持稳定，内部实现允许演化；不要让内部重构迫使调用方整体改写。
4. API 应对 AI 友好：参数名清晰统一、字段尽量少、嵌套尽量浅、单次动作语义单一。
5. 状态设计必须显式表达：状态集合、流转规则、动作约束。
6. 关键模型、关键状态、关键业务动作必须有唯一入口，避免多点写入和隐式改写。
7. 后端实现优先关注数据一致性和状态一致性，不以“代码没报错”替代一致性判断。

### 7.5 检查清单

应至少包含三段检查：

#### 开发前

- 这次改动属于核心规则，还是边界适配？
- 是否会改变对外契约？
- 当前动作是否语义单一？
- 是否已经明确状态集合、状态入口和动作约束？

#### 开发中

- 是否把外部协议字段、第三方返回结构或存储细节污染到了核心模型？
- 是否有多个模块都能写同一关键状态？
- 当前接口是不是已经过长、过深、过度嵌套？
- 是否通过隐式副作用完成状态变化？

#### 开发后

- 是否能清楚回答“谁能改这个状态、在哪里改、按什么规则改”？
- 是否切断了不必要的变化传播？
- 调用方是否仍能通过稳定契约使用该能力？
- 一致性约束是否仍然清楚且可验证？

### 7.6 反模式

应明确列出的后端反模式至少包括：

- 业务规则直接依赖外部协议格式
- 在多个 service、repository、handler 中直接修改同一核心状态
- 一个接口同时承载多个动作语义
- 为了“一次查全”构造巨大且多层嵌套的输入输出结构
- 用隐式读写、魔法副作用或跨模块偷改状态完成业务逻辑

### 7.7 示例要求

示例只保留少量、通用、跨项目可迁移的内容，例如：

- 什么时候应该拆分一个过长接口
- 什么时候需要建立唯一状态入口
- 什么时候外部协议映射应停留在适配层

## 8. 人工确认边界与自动执行边界

两个 Skill 都应显式区分两类边界：

### 8.1 必须人工确认

- 会改变用户体验和产品语义的改动
- 会改变系统边界和关键对象定义的改动
- 会改变外部契约和核心状态规则的改动

### 8.2 可按 Skill 直接执行

- 结构收敛
- 反模式修正
- 局部拆分与职责整理
- 契约内补实现
- 可复用实现的落地与替换

## 9. Skill 与项目文档的关系

这两个 Skill 的定位是：

- 通用开发约束
- 工程质量判断器
- 反模式拦截器

而不是：

- 项目规格文档入口
- 项目事实来源
- 架构现状镜像

项目文档负责描述“这个项目当前是什么”，Skill 负责约束“开发时应该怎么做才不容易失控”。

## 10. 后续落地建议

在此设计稿获批后，再进入实际 Skill 编写。实现顺序建议为：

1. 先写 `frontend-development`
2. 再写 `backend-development`
3. 最后再回收根 `AGENTS.md` 中不适合保留的细规则

实现阶段需继续遵守：

- Skill 内容保持通用
- 引用材料只补重内容
- 示例少而有效
- 不把项目 spec 复制进 Skill
