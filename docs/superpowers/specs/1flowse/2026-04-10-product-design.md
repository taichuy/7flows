# 1Flowse 产品设计讨论稿

日期：2026-04-10
状态：已审阅，已进入需求与架构拆解

## 1. 产品定位

1Flowse 的定位是：一个轻量、可扩展、以标准 Agent API 发布为中心的 AI 工作流平台。

它在 P1 阶段不应被定义为：
- 聊天优先的 AI 产品
- 通用型低代码平台
- 完整企业级多租户 PaaS

它要解决的核心问题是：让团队把一个 AI Flow 编排出来，并以稳定的外部接口形式发布出去，让 Codex、Claude Code、本地 Agent 或其他客户端可以直接调用。

在这个定位下：
- `Flow` 是核心资产
- `Publish Endpoint` 是核心交付物
- `State Model` 是跨会话记忆与上下文持久化层
- `Plugin Framework` 是扩展基础设施
- 平台内对话框与页面能力都是辅助面，不是产品中心

## 2. 产品优先级

本次讨论确认的优先级如下：

1. `标准 Agent 兼容发布` 是产品主目标。
2. `状态与记忆管理` 是产品竞争力。
3. `插件扩展体系` 是架构支撑层。

因此，1Flowse 应被描述为一个 `以发布为优先的 AI 工作流平台`，而不是功能堆叠式 AI Builder。

## 3. 核心产品原则

- `工作流优先`：用户的核心操作对象始终是 Flow。
- `发布优先`：一个 Flow 只有能稳定对外暴露后，才算真正完成。
- `运行时优先于编辑器元数据`：平台能力必须建立在可恢复、可持久化、可追踪的运行时模型之上。
- `状态必须结构化`：记忆、上下文、业务状态不应只是散落变量，而应是可管理的数据结构。
- `可观测性是产品界面的一部分`：日志、工具调用、节点输入输出、token 消耗都必须作为用户看得见的能力存在。
- `权限由后端统一治理`：资源权限与路由权限都应落在统一后端策略层，不依赖前端拼接逻辑。

## 4. P1 目标范围

P1 是一个面向团队协作的 AI 工作流平台。

P1 只需要证明一个闭环：

1. 团队可以创建应用。
2. 应用中可以编排包含多个节点的 Flow，包括多个 LLM 节点。
3. Flow 可以在调试阶段看到节点级运行状态。
4. Flow 可以发布成标准 Agent 友好的 API。
5. 外部客户端可以通过稳定接口调用该 Flow。
6. 运行时可以正确处理工具调用、暂停、回调、恢复与 checkpoint。
7. 状态模型可以在多次调用之间保存上下文与记忆。

如果 P1 能让团队把应用建出来、发出去、跑起来、查得到、控得住，就算达成目标。

## 5. 核心领域对象

### 5.1 Team Workspace

`Team Workspace` 是 P1 唯一的一等空间对象。

以下资源都归属到团队空间：
- Flow
- Publish Endpoint
- State Model
- State Data
- External Data Source
- Plugin Config
- 日志与监控视图
- 后台路由与应用配置

### 5.2 Tenant

`Tenant` 在 P1 里仅作为未来扩展预留字段存在。

它不参与当前的产品层空间设计、权限设计和管理设计。

### 5.3 Application

`Application` 是工作台中用户直接管理的交付单元。

一个应用包含：
- 一个或多个 Flow
- 发布配置
- 运行日志
- 监控视图
- 状态与数据绑定
- 应用级设置

应用是用户组织 AI 能力、交付 AI 能力的主要容器。

### 5.4 Flow

`Flow` 是最核心的可编辑资产。

一个 Flow 是可版本化的工作流定义，包含：
- 节点
- 连线
- 变量定义
- 输入输出契约
- 状态读写行为
- 发布相关设置

### 5.5 Flow Run

`Flow Run` 是一次 Flow 的运行实例。

它承载：
- 调用输入
- 运行时上下文
- 状态引用
- 发布协议相关会话信息
- checkpoint
- 最终输出

### 5.6 Node Run

`Node Run` 是某个节点在一次 Flow Run 中的执行实例。

这个对象必须是一等概念，因为多 LLM 节点、工具调用暂停、回调恢复都属于产品核心语义，而不是实现细节。

### 5.7 State Model

`State Model` 是结构化持久化层，用于承接：
- 跨会话记忆
- 业务状态持久化
- 上下文注入
- 基础运营与管理数据

它不应被定位成通用低代码数据平台。

### 5.8 Publish Endpoint

`Publish Endpoint` 是 Flow 的对外交付物。

用户不是只在平台内部运行 Flow，而是需要把它稳定发布为可被外部 Agent 和客户端调用的接口。

### 5.9 Plugin

`Plugin` 是扩展能力载体，覆盖：
- 模型供应商接入
- 节点类型扩展
- 数据源扩展
- 发布协议适配扩展

## 6. 执行模型与节点状态

1Flowse 的编排不能被视为“简单顺序执行”。它必须支持暂停、恢复、回调、checkpoint 持久化与异常恢复。

### 6.1 运行时对象

- `Flow Run`：一次完整工作流执行会话
- `Node Run`：一个节点在某次执行中的状态载体
- `Checkpoint`：运行时关键边界的持久化恢复点
- `Callback Task`：等待工具结果、异步结果或人工输入的待恢复任务

### 6.2 Flow Run 状态

- `queued`
- `running`
- `waiting_callback`
- `waiting_human`
- `paused`
- `succeeded`
- `failed`
- `cancelled`

### 6.3 Node Run 状态

- `pending`
- `ready`
- `running`
- `streaming`
- `waiting_tool`
- `waiting_callback`
- `waiting_human`
- `retrying`
- `succeeded`
- `failed`
- `skipped`

### 6.4 必须明确支持的运行语义

- 一个 Flow 中可以包含多个 LLM 节点。
- 一个 LLM 节点既可能产出文本，也可能产出结构化结果、工具调用请求或控制信号。
- 当 LLM 节点触发工具调用时，运行时应暂停当前节点，而不是立刻推进到下一个节点。
- 节点暂停后，系统必须先落 checkpoint，再进入等待外部输入或 callback 完成状态。
- callback 数据回填后，必须先恢复当前被暂停节点，再继续推进下游节点。
- 只有当前节点进入稳定完成态后，下游节点才允许开始执行。
- 节点之间的数据传递应基于统一内部对象协议，而不是松散文本拼接。

这些能力之所以是一等能力，是因为标准 Agent 协议天然要求支持工具调用、后续查询、多步执行和恢复执行。

## 7. 产品模块与边界

### 7.1 Workflow Studio

负责：
- 节点可视化编排
- Flow 版本编辑
- 变量与契约定义
- 发布配置
- 调试入口

### 7.2 Runtime Orchestrator

负责：
- 节点调度
- 状态传递
- 工具调用暂停与恢复
- checkpoint 持久化
- callback 恢复
- 异常处理
- 流式输出

### 7.3 Publish Gateway

`Publish Gateway` 是产品最重要的差异化层。

P1 推荐采用：
- 一套统一内部运行协议
- 多个对外适配层

这样内部运行模型��持统一，对外则可以逐步映射成不同 Agent 友好的协议形式，而不会把内部逻辑拆碎。

### 7.4 State Service

负责：
- 状态模型定义
- 关系定义
- CRUD 自动生成
- 上下文注入策略
- 运行时状态访问
- 基础运营态数据管理

它的职责是支撑记忆与状态，而不是演化成一个完整低代码平台。

### 7.5 Plugin Framework

负责：
- 插件能力声明
- 生命周期管理
- 注册与发现
- 模型供应商扩展
- 节点扩展
- 数据源扩展
- 发布适配扩展

### 7.6 Access Control

`Access Control` 负责在后端统一治理资源权限与路由权限。

### 7.7 Admin Console

`Admin Console` 在 P1 只提供最小必要能力：
- 状态数据管理
- 查询与表格视图
- 配置界面
- 资源级运营管理

P1 不做完整页面搭建器。

## 8. 团队、资源与权限模型

权限设计必须同时覆盖资源对象和路由访问。

### 8.1 一等资源

P1 至少需要对以下资源做授权控制：
- `Flow`
- `Publish Endpoint`
- `State Model`
- `State Data`
- `External Data Source`
- `Plugin Config`
- `Admin Route / View`

### 8.2 动作模型

至少支持以下动作维度：
- `view`
- `edit`
- `publish`
- `manage`
- `use`
- `configure`

### 8.3 基础可见性策略

P1 需要支持明确但不复杂的策略，例如：
- 仅创建者可见
- 团队成员可见
- 团队成员可修改
- 指定角色可管理
- 指定角色可发布或调用

### 8.4 外部数据源权限

`External Data Source` 是敏感资源，必须单独建模权限。

至少要区分：
- 谁可以创建或配置数据源
- 谁可以查看数据源元信息
- 谁可以在 Flow 或 State Model 中使用数据源
- 谁可以管理连接凭据

### 8.5 路由访问策略

路由不能绕过资源授权。

一个路由只有在以下两个条件同时满足时才可访问：
- 用户拥有该路由访问权限
- 用户拥有该路由绑定资源的访问权限

这样可以保证后台页面、数据视图和资源对象都落在同一套后端权限体系中。

## 9. 交互设计与信息架构

### 9.1 顶层导航

平台顶层导航固定为：
- `工作台`
- `团队`
- `后台设置`

含义如下：
- `工作台`：应用管理入口
- `团队`：成员、角色、资源授权与协作访问管理
- `后台设置`：模型供应商、插件、数据源及平台级配置

应用不应成为顶层导航类目，而应作为工作台中的资源对象存在。

### 9.2 工作台

工作台负责：
- 创建应用
- 浏览应用
- 搜索应用
- 查看最近访问
- 查看应用发布状态概览

### 9.3 应用工作区

进入某个应用后，界面采用左右结构：
- 左侧为功能导航
- 右侧为当前功能的主工作区

### 9.4 应用内左侧导航

应用内至少包含以下栏目：
- `画布编排`
- `API 文档`
- `对话日志`
- `监控`
- `状态与数据`
- `应用设置`

### 9.5 画布编排

`画布编排` 基于 `xyflow` 实现无限画布，用于：
- 节点编排
- 分支控制
- 变量连线
- 状态读写
- 发布行为配置

### 9.6 API 文档

`API 文档` 是一等页面，因为“可发布、可接入”是产品承诺的核心。

页面需要展示：
- 接口形态
- 鉴权方式
- 请求格式
- 响应格式
- callback 行为
- 示例调用方式

### 9.7 对话日志

`对话日志` 是核心产品能力，不只是消息记录页。

每条日志至少要包含：
- 对话内容
- 每次工具调用
- 节点级输入输出
- 暂停与恢复事件
- callback 过程
- 失败信息
- 最终结果

这个页面既承担调试功能，也承担审计、追踪和用户信任建立的作用。可以参考成熟产品中“可观测对话”体验的优点，但要更贴合多节点 Flow 运行模型。

### 9.8 监控

`监控` 是一等页面，不只是内部运维后台。

P1 用户关心的指标至少包括：
- 某次对话消耗了多少 `token`
- 今日请求次数
- 应用调用总量
- 成功率与失败率
- 平均响应时长
- 工具调用次数
- 各模型供应商消耗分布
- 高频节点
- 失败高发节点
- 近期异常会话数

这些指标服务的是用户决策问题：成本、稳定性、调用量和热点问题。

### 9.9 状态与数据

`状态与数据` 页面用于：
- 状态模型管理
- 关系管理
- 数据浏览
- CRUD 操作
- 数据源映射
- 上下文检查

它服务于应用记忆与状态，不承担通用低代码平台角色。

### 9.10 AI Assistant 辅助面

AI 辅助能力可以以侧栏或上下文助手形式存在，但始终是次要入口。

它的职责是辅助：
- 节点配置
- Flow 编辑
- 前端代码片段生成
- 模型说明
- 状态设计建议

它必须始终依附当前应用上下文，而不是变成一个脱离上下文的大聊天框。

## 10. P1 应包含的能力

P1 应至少包含：
- 团队空间模型
- 角色与权限控制
- 工作流画布编辑
- 单个 Flow 中支持多个 LLM 节点
- 工具调用暂停与恢复
- callback 恢复
- checkpoint 持久化
- 面向标准 Agent 的发布接口
- 基础状态模型管理
- 状态模型 CRUD 自动生成
- 基础后台与查询视图
- 内置模型供应商插件
- 内置节点插件
- 应用级日志与监控

## 11. P1 明确不做的内容

P1 应明确避免：
- 聊天优先的产品定位
- 复杂的可视化页面搭建器
- 完整的企业级租户治理与组织治理
- 个人空间模型
- 完整插件市场
- 通用低代码平台定位
- 把外部业务数据库替代作为核心产品卖点

## 12. 成功标准

如果 P1 成功，应至少满足以下结果：

- 团队可以在团队空间中创建并管理应用
- 用户可以编排包含多个 LLM 节点和其他辅助节点的 Flow
- Flow 可以通过统一内部协议和对外适配层发布为可调用接口
- 外部客户端可以稳定调用已发布 Flow
- 运行时可以正确处理工具调用暂停、callback、恢复与 checkpoint 恢复
- 对话日志可以清晰展示对话内容、工具调用和节点级执行细节
- 监控可以展示用户真正关心的成本、调用量与健康度指标
- 结构化状态可以支撑跨会话记忆与上下文延续
- 权限体系可以稳定控制 Flow、数据、路由与外部集成访问

## 13. 一句话验收标准

当一个团队能够把 AI Flow 编排出来、稳定发布成外部 Agent 可调用能力，并在整个运行生命周期中获得可追踪的日志、监控、状态和权限控制时，1Flowse 的 P1 就达成了目标。
