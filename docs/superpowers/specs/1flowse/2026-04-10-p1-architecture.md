## 1. 文档目标

本文档用于明确 1Flowse P1 的技术架构边界，回答以下问题：

- 前端、后端、运行时、发布层如何分工
- 什么能力应该在 P1 采用模块化单体，什么能力应逻辑拆分
- PostgreSQL、Redis、RustFS 各自承担什么职责
- 多 LLM 节点、工具调用、callback、checkpoint 如何落到运行时模型
- AI 友好的页面结构协议、嵌入式前端与反向代理如何落地
- 插件体系、权限体系、日志监控如何嵌入主系统而不把系统做重

## 2. 架构目标

P1 架构目标如下：

1. 保持整体足够轻，不提前进入微服务复杂度。
2. 保证运行时对多节点、暂停恢复、callback 的可靠支持。
3. 让对外发布能力和内部编排运行能力使用统一核心模型。
4. 让状态与记忆能力成为标准支撑层，而不是外挂模块。
5. 保证日志、监控、权限和数据源治理可以作为一等能力沉入后端。
6. 为后续插件扩展和更多协议适配预留清晰接口。
7. 让未来的 AI 页面生成建立在稳定的页面描述协议之上，而不是任意前端代码执行之上。

## 3. 架构原则

- `模块化单体优先`：P1 采用单仓、单套核心后端、逻辑拆分的方式，不直接拆微服务。
- `控制面与运行面逻辑分离`：虽然可以先部署在同一后端体系中，但接口边界必须明确区分。
- `统一内部协议优先`：先定义 Flow 运行与节点交互的内部协议，再做外部适配。
- `默认页面协议优先`：页面默认以结构化页面协议描述，再由平台渲染。
- `持久化状态优先`：所有暂停、恢复、回调都必须建立在可持久化运行状态上。
- `PostgreSQL 作为事实来源`：关键元数据和可恢复运行状态以 PostgreSQL 为主。
- `Redis 作为热态协调层`：用于缓存、短期状态、事件分发、队列协调，而不是长期主存储。
- `对象存储承接大对象`：文件、导出物、较大日志片段和后续扩展资产存入 RustFS。
- `嵌入式前端作为高自由度通道`：复杂前端通过嵌入式应用接入，不作为默认页面生成方式。

## 4. 总体架构

P1 推荐采用：

`前端应用 + Rust 模块化单体后端 + 独立运行时 Worker + PostgreSQL + Redis + RustFS`

逻辑上分为两大平面：

- `控制面`
  - 工作台
  - 团队与权限
  - 后台设置
  - 应用配置
  - Flow 编辑
  - 状态模型管理
  - 数据源管理
- `运行面`
  - 发布接口
  - Flow 执行
  - 节点调度
  - callback 恢复
  - 日志事件流
  - 指标聚合

建议的部署形态如下：

- `web`
  - 前端工作台与应用界面
- `api-server`
  - 控制面 API
  - 发布入口 API
  - callback 接口
  - 日志与监控查询 API
- `runtime-worker`
  - Flow 调度、节点执行、恢复执行、异步任务处理
- `postgres`
  - 元数据与持久化运行状态
- `redis`
  - 热态缓存、队列、分发、会话级协调
- `rustfs`
  - 文件与大对象存储

## 5. 前端架构

### 5.1 前端技术方向

前端采用 `pnpm` 管理，建议使用 monorepo 结构，便于在应用端、共享组件和协议定义之间保持一致。

建议分层为：
- `apps/web`
  - 平台主前端
- `packages/ui`
  - 通用 UI 组件
- `packages/flow-schema`
  - Flow 节点与连线的前端结构定义
- `packages/page-protocol`
  - AI 友好的页面结构协议定义
- `packages/page-runtime`
  - 动态页面渲染运行时
- `packages/api-client`
  - 控制面 API 与运行查询 API 客户端
- `packages/embed-sdk`
  - 嵌入式前端应用使用的 SDK
- `packages/shared-types`
  - 与后端协议对齐的轻量共享类型

### 5.2 前端界面分区

前端顶层固定为：
- `工作台`
- `团队`
- `后台设置`

进入应用后，采用左右结构：
- 左侧：应用导航
- 右侧：主工作区

应用内页面至少包括：
- `画布编排`
- `API 文档`
- `对话日志`
- `监控`
- `状态与数据`
- `应用设置`

### 5.3 画布层

画布基于 `xyflow` 构建，负责：
- 节点与连线展示
- 节点参数编辑入口
- 分支与流程结构表达
- 节点状态高亮
- 发布配置入口

前端画布层只负责编辑体验和运行态展示，不负责真正的执行逻辑。

### 5.4 前端数据获取策略

前端应区分两类数据：

- `控制面数据`
  - 应用、Flow、权限、数据源、状态模型等
- `运行面数据`
  - Run 状态、Node Run 状态、对话日志、指标、流式事件

运行态数据建议优先通过：
- 普通查询接口读取历史
- 流式通道或轮询读取运行中状态

### 5.5 前端三层模型

P1 前端应采用三层模型：

- `Headless Plugin`
  - 只负责扩展后端运行时能力
  - 不作为主页面扩展机制
- `AI 友好的页面结构协议`
  - 平台默认页面方案
  - 用户创建路由和页面、绑定数据源后，AI 或用户配置生成页面结构协议
  - 前端运行时负责将协议渲染为动态页面
- `Embedded App`
  - 用户自定义开发的独立前端系统
  - 通过 SDK、同源代理和统一权限接入平台

### 5.6 AI 页面生成链路

建议的页面生成链路为：

1. 用户在后台创建路由和页面。
2. 用户为页面选择数据源、状态模型或 CRUD 资源。
3. 系统组装 `PageGenerationContext`。
4. AI 根据上下文输出 `Page Definition`，其核心内容是页面结构协议。
5. 页面定义持久化到后端。
6. 前端运行时根据页面结构协议渲染页面。

`PageGenerationContext` 至少应包含：
- 路由信息
- 页面目标
- 数据源与资源名
- 字段与关系结构
- 已生成 CRUD 能力
- 可用区块能力
- 权限边界

### 5.7 嵌入式前端接入

当默认页面协议无法表达复杂业务前端时，允许用户接入 `Embedded App`。

推荐模式：
- 用户在外部开发完整前端
- 构建产物部署到前端服务器或对象存储
- 平台通过同源反向代理挂接到应用路由下
- 平台通过 SDK 注入上下文、鉴权和 API 能力

P1 不建议采用“数据库直接存储 JSX/TSX 并在线执行”的方案。

## 6. 后端总体架构

### 6.1 Rust Workspace 组织方式

后端建议采用 Rust workspace，按模块拆 crate，而不是一开始拆成多个独立仓库。

建议的结构方向：

- `apps/api-server`
  - HTTP 入口，承接控制面 API、发布入口和 callback 接口
- `apps/runtime-worker`
  - 运行时任务消费、节点调度、恢复执行
- `crates/domain`
  - 核心领域模型，如 Application、Flow、Run、NodeRun、StateModel、Permission
- `crates/control-plane`
  - 工作台、团队、后台设置等控制面服务
- `crates/page-service`
  - 页面定义、路由管理、页面生成上下文与嵌入式页面登记
- `crates/runtime-core`
  - 执行引擎、状态机、调度器、checkpoint 逻辑
- `crates/publish-gateway`
  - 统一内部协议与外部协议适配层
- `crates/plugin-framework`
  - 插件注册、能力声明、调用抽象
- `crates/access-control`
  - 角色、资源授权、路由授权
- `crates/storage-pg`
  - PostgreSQL 仓储实现
- `crates/storage-redis`
  - Redis 缓存、队列、事件协调
- `crates/storage-object`
  - RustFS 对象存储封装
- `crates/observability`
  - 运行事件、日志聚合、指标采集

### 6.2 模块化单体而非微服务

P1 不建议把控制面、发布网关、运行时、权限、日志系统拆成独立微服务。

原因：
- 业务还在快速收敛阶段，接口会持续变化
- 运行时、日志、权限和发布层强耦合
- 当前更需要一致的运行语义，而不是独立部署复杂度
- 后续若调用量上升，先把 `runtime-worker` 与 `api-server` 横向扩容即可

## 7. 控制面与运行面边界

### 7.1 控制面职责

控制面主要负责：
- 应用管理
- Flow 定义与版本管理
- 路由与页面定义管理
- 状态模型管理
- 数据源配置
- 团队成员与角色管理
- 后台设置
- 发布配置

控制面 API 的特点是：
- 面向人类操作界面
- 以 CRUD、配置和查询为主
- 数据一致性要求高

### 7.2 运行面职责

运行面主要负责：
- 接收已发布应用的调用请求
- 创建 Flow Run
- 调度节点执行
- 处理工具调用与 callback
- 持久化运行状态
- 产生日志事件
- 汇总运行指标

运行面 API 的特点是：
- 面向 Agent 与程序调用
- 以执行、恢复、流式返回为主
- 强调低负担、稳定性和幂等性

## 8. 统一内部协议

P1 需要先定义统一内部协议，再把它映射成外部调用形式。

内部协议至少应覆盖以下对象：
- `InvokeRequest`
- `InvokeContext`
- `FlowRunSnapshot`
- `NodeInput`
- `NodeOutput`
- `ToolCallRequest`
- `CallbackResumeRequest`
- `StateReadResult`
- `StateWritePatch`
- `InvokeResponse`

设计要求：
- 节点之间传递统一对象，不直接依赖某厂商响应格式
- 运行时与发布层共享同一套中间语义
- 对外适配层只负责映射，不负责改写核心执行逻辑

## 9. 运行时架构

### 9.1 核心运行对象

运行时一等对象包括：
- `Flow Run`
- `Node Run`
- `Checkpoint`
- `Callback Task`
- `Run Event`

### 9.2 调度模型

建议采用事件驱动的调度模型：

1. 发布入口接收到调用请求。
2. 创建 `Flow Run` 与初始 `Run Event`。
3. 调度器按依赖关系推进可执行节点。
4. 节点执行过程中的状态变化追加为事件。
5. 若节点进入 `waiting_tool` 或 `waiting_callback`，则落 checkpoint。
6. callback 到来后，恢复对应节点并继续推进。
7. Flow 完成后写入最终状态与输出。

### 9.3 为什么需要 Run Event

`Run Event` 是日志页和监控页的共同基础。

它至少需要记录：
- Run 创建
- Node 开始执行
- Node 输出
- Tool Call 发起
- Tool Call 返回
- Pause
- Resume
- Callback 收到
- Failure
- Completion
- Token 统计

这样做的好处是：
- 日志页有完整轨迹
- 监控页可以按事件聚合
- 调试页可以回放关键路径

### 9.4 状态机要求

Flow Run 与 Node Run 状态机应与设计稿保持一致：

- Flow Run：
  - `queued`
  - `running`
  - `waiting_callback`
  - `waiting_human`
  - `paused`
  - `succeeded`
  - `failed`
  - `cancelled`

- Node Run：
  - `pending`
  - `ready`
  - `running`
  - `streaming`
  - `waiting_tool`
  - `waiting_callback`
  - `waiting_human`
  - `retrying`
  - `succeeded`
  - `failed`
  - `skipped`

## 10. 存储分层

### 10.1 PostgreSQL

PostgreSQL 是系统事实来源，负责持久化：
- 团队、成员、角色、权限
- 应用与 Flow 元数据
- 应用路由与页面定义
- Flow 版本
- 发布配置
- 状态模型定义
- 状态数据
- 数据源元信息
- Flow Run
- Node Run
- Checkpoint 元信息
- Run Event 索引与核心事件

原则：
- 只要涉及恢复执行、权限判断、结构化查询，就优先落 PostgreSQL

### 10.2 Redis

Redis 负责热态与协调能力：
- 运行中会话的短期缓存
- 调度队列
- callback 等待索引
- 分布式锁
- 热点指标聚合
- 实时事件推送缓冲

原则：
- Redis 不作为唯一事实来源
- Redis 中的数据应允许丢失后重建

### 10.3 RustFS

RustFS 用于存储大对象或文件类数据，例如：
- 上传文件
- 大型日志片段
- 导出归档
- 后续插件资产
- 嵌入式前端构建产物
- 页面生成产物或附件

原则：
- RustFS 不承接核心结构化业务查询

## 11. 状态与数据建模架构

P1 的状态层不是独立数据库产品，而是 Flow 的结构化记忆层。

架构要求：
- 状态模型元数据存 PostgreSQL
- 状态记录优先存 PostgreSQL
- 状态访问通过统一状态服务完成
- 运行时不得直接绕过状态服务访问底层表

这样做的目的：
- 保持权限与审计一致
- 为后续自动生成 CRUD 和后台页面提供统一入口
- 保证状态数据可被安全注入上下文

## 12. 数据源架构

外部数据源是高风险资源，建议拆成两层：

- `Data Source Definition`
  - 连接类型、名称、描述、可见性、拥有者
- `Data Source Secret`
  - 凭据、密钥、令牌等敏感信息

架构要求：
- 元信息与敏感信息分离存储
- 使用前必须经过权限检查
- Flow 节点与状态服务均通过统一数据源访问层调用

## 13. 页面与嵌入式前端架构

### 13.1 页面定义

平台默认页面不直接存储任意可执行前端代码，而是存储 `Page Definition`。

`Page Definition` 应至少包含：
- 路由信息
- 页面元数据
- 区块树
- 数据绑定
- CRUD 动作绑定
- 权限绑定
- 布局信息

其中“区块树 + 数据绑定 + 动作绑定”构成 `AI 友好的页面结构协议`。

### 13.2 动态页面渲染

前端运行时负责：
- 加载页面定义
- 校验协议版本
- 将区块树映射到平台组件
- 绑定 CRUD 与数据请求
- 在页面层执行权限包裹

这让 AI 或用户生成页面时，不需要直接操纵完整 React 应用骨架。

### 13.3 Embedded App

`Embedded App` 用于承接高自由度自定义系统。

它的接入方式建议为：
- 后台登记嵌入式页面元数据
- 指定挂载路由
- 通过同源反向代理将请求转发到目标前端服务或静态产物
- 注入平台 SDK 所需上下文

需要统一注入的上下文至少包括：
- `teamId`
- `applicationId`
- 当前用户信息
- API 基础地址
- 鉴权令牌或会话上下文
- 当前路由与 locale

### 13.4 为什么不采用直接存储 JSX

不推荐将任意 JSX/TSX 直接存数据库并在线执行，原因是：
- 安全边界差
- 平台升级后兼容性差
- 构建与依赖不可控
- 权限和数据访问容易绕过统一治理
- 调试、回滚、审计成本高

因此建议：
- 默认页面走页面结构协议
- 复杂页面走嵌入式前端

## 14. 插件架构

P1 插件体系建议先采用 `进程内插件 + 清晰能力协议` 模式，而不是一开始做远程插件沙箱。

P1 支持的插件类型：
- `Provider Plugin`
- `Node Plugin`
- `Data Source Plugin`
- `Publish Adapter Plugin`

这些插件默认应是 `Headless Plugin`，即：
- 扩展后端能力
- 提供配置 schema
- 不把主页面扩展能力做成前端插件系统

插件框架需要提供：
- 插件清单
- 能力声明
- 注册与发现
- 调用抽象
- 生命周期钩子

P1 先解决：
- 内置插件如何统一接入
- 节点与供应商如何通过同一协议接入运行时

P1 暂不解决：
- 第三方插件市场
- 复杂插件发布流程
- 独立插件运行沙箱体系

## 15. 权限与路由治理架构

权限层必须在后端统一实现，不能散落在各控制器与页面中。

建议采用：
- `资源授权`
- `动作授权`
- `路由绑定资源`

执行顺序建议为：

1. 校验用户是否属于当前团队空间。
2. 校验用户是否拥有目标路由访问资格。
3. 校验用户是否拥有路由绑定资源的目标动作权限。
4. 通过后才允许访问实际业务处理器。

这样可以保证：
- 页面不会绕过资源权限
- 数据源等敏感对象有统一保护
- 后续插件页面也可以复用同一套权限检查

## 16. 日志与监控架构

### 16.1 日志

日志页依赖运行事件流。

建议架构：
- 节点执行过程持续产出 `Run Event`
- 关键事件持久化到 PostgreSQL
- 热态更新先写 Redis 缓冲，再异步或同步归档

日志查询应支持：
- 按应用查
- 按 Flow 查
- 按会话查
- 按状态查
- 按错误查

### 16.2 监控

监控页不要求 P1 建立复杂时序平台，但需要有可聚合指标。

建议做法：
- 从 Run Event 中提取聚合指标
- 定时或实时写入指标聚合表
- 前端按日、按应用、按 Flow 展示

P1 重点指标：
- token 使用量
- 请求次数
- 成功率
- 失败率
- 平均延迟
- 工具调用次数
- 供应商消耗分布

## 17. 发布网关架构

发布网关负责把已发布 Flow 暴露给外部调用方。

它需要解决：
- 请求鉴权
- 请求映射到统一内部协议
- 调用运行时
- 流式返回或普通返回
- callback 路径接入
- 响应映射

网关不应承担：
- Flow 编辑逻辑
- 权限配置逻辑
- 节点内部执行逻辑

## 18. 部署与演进建议

### 18.1 P1 部署建议

P1 建议先部署为：
- 1 个前端应用
- 1 个 `api-server`
- 1 个 `runtime-worker`
- 1 套 PostgreSQL
- 1 套 Redis
- 1 套 RustFS

这已经足够支撑当前需求，并保留未来横向扩展空间。

### 18.2 后续演进方向

当系统增长后，可以按需演进：
- 将 `runtime-worker` 独立扩容
- 将发布入口与控制面入口分离部署
- 为插件执行引入独立沙箱
- 为监控引入更强时序与分析系统
- 为对象存储引入生命周期与归档策略

## 19. 当前架构结论

P1 最合适的方案不是微服务堆叠，而是：

`前端 pnpm monorepo + 页面结构协议驱动的动态页面运行时 + 嵌入式前端接入能力 + Rust 模块化单体后端 + 独立运行时 Worker + PostgreSQL / Redis / RustFS 分层存储 + 统一内部协议 + Headless 插件扩展接口`

这个方案最符合当前产品目标：
- 足够轻
- 能支撑复杂运行语义
- 对外发布能力清晰
- 权限、日志、状态和扩展都能统一落位
