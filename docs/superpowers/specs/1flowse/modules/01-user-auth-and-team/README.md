## 本模块范围

- 用户登录、退出、改密
- 单一 `Team Workspace` 初始化与团队基础配置入口
- 成员账号创建、停用、角色绑定入口
- 会话基础能力
- 与权限模块衔接的角色骨架和授权载体定义

## 已确认

- P1 唯一一等空间对象是 `Team Workspace`。
- `Tenant` 在 P1 仅作预留字段，不进入当前空间与登录设计。
- P1 暂不做个人工作空间。
- P1 登录主方式为 `账号 + 密码`。
- 默认允许使用 `邮箱` 直接作为登录账号。
- `电话` 字段为非必填，且默认不启用为登录账号；后端需支持按用户配置决定哪些字段可作为登录标识。
- P1 暂不做第三方登录。
- P1 不强制邮箱验证通过后才可登录。
- P1 不开放用户自助注册。
- 单一 `Team Workspace` 为初始化产生的引导型业务空间，而不是 `root` 的权限边界。
- 初始化流程必须幂等，重复执行不能生成第二个 `Team Workspace`、第二个 `root` 账号或重复内置角色模板。
- 仅 `.env` 中配置的默认账号拥有 `root` 角色。
- P1 预置应用级 `root` 与空间级 `admin`、`manager` 三个内置角色模板。
- `root` 是应用级内置超管，权限默认全开且不可修改，不以 `Team Workspace` 为作用域边界。
- `admin` 不是超管，而是默认拥有全部权限点的可编辑角色。
- `manager` 仅管理业务资源，不管理成员和团队配置。
- `admin`、`manager` 属于 `Team Workspace` 角色模板，当前默认服务于初始化空间；未来若扩展新空间或子租户，可按空间独立配置。
- 用户可同时绑定多个角色。
- 成员账号通过后台创建，不走邀请链接或邀请码流程。
- 新创建成员默认绑定 `manager` 角色，创建后可再增删其他角色。
- 创建成员时手动设置初始密码，并线下告知成员。
- 成员首次登录不强制改密，但可在设置页自行修改密码。
- 用户可在前端设置自己的默认展示角色，但该设置只影响默认界面视角，不改变后端真实权限集合。
- 用户自己仅可修改个人资料、登录密码与默认展示角色，不能修改自身角色或权限。
- P1 不提供自助密码找回。
- 普通成员忘记密码时，由具备成员管理权限的 `admin/root` 在后台直接重置密码，不走邮件找回或自助重置。
- 后台重置密码后，用户可直接使用新密码登录，不强制首次登录改密。
- 后端仍保留针对 `.env` 预置 `root` 账号的密码重置脚本，作为超管兜底链路。
- 会话策略为 `服务端 Session + HttpOnly Cookie`，采用 `7 天滑动续期`。
- 成员离职或停用时采用禁用账号，保留历史数据与归属关系。
- `admin` 可管理成员、禁用账号、维护团队配置。
- `manager` 可创建、编辑、删除自己的应用。
- `root` 默认可访问并接管初始化 `Team Workspace`，同时保留未来跨空间治理与紧急运维能力。
- P1 团队基础配置边界仅包含团队名称、团队 Logo/图标、团队简介。
- 默认语言、时区等展示偏好不纳入 P1 团队基础配置，继续归于个人设置或后续系统设置讨论。
- 权限采用“角色 + 权限点”模型。
- 权限结构分为应用级与空间级两层。
- 权限点只挂在角色上。
- 用户绑定多个角色。
- 后端鉴权顺序为：先判应用级 `root`，再判当前空间内角色权限点并集。
- 前端基于用户当前选中角色做入口与页面展示裁剪，但前端裁剪不作为安全边界。
- `root` 默认进入全局管理视角；非 `root` 用户按当前选中角色切换界面视角。
- 当前已明确的权限类型包括：
  - 路由权限
  - 权限配置权限
  - 表级查看/编辑/删除权限，并区分“自己的数据 / 全部数据”
- 当前关注的权限点范围聚焦为：
  - 应用级全局治理能力：仅 `root` 拥有全局放行、空间接管、初始化接管、全局权限配置、紧急运维能力
  - 成员管理：查看成员、创建成员、编辑基础信息、启用/禁用账号、重置密码、绑定/解绑角色
  - 团队配置管理：查看并修改当前 `Team Workspace` 基础配置
  - 角色与权限配置：查看角色、创建空间自定义角色、编辑空间角色模板、为角色绑定权限点，但不能创建第二个 `root` 或修改 `root` 本身
  - 管理应用
  - 管理路由
  - 开放表增删改查权限（区分“仅自己 / 所有数据”）
- 系统自建表仅开放用户表和权限表作为可配权限对象。
- 应用编排等复杂内部表在 P1 不注册为通用权限点。
- 默认拥有相关权限点的角色即可执行对应管理动作，不再额外对某个角色做硬编码特判。
- P1 默认能力矩阵收敛为：
  - `root`：应用级全局放行，可接管任意空间与全部配置
  - `admin`：当前空间内可管理成员、团队配置、空间角色与权限，以及全部业务资源
  - `manager`：默认只管理自己负责的应用、路由与业务数据
  - 普通用户自助能力：仅修改个人资料、登录密码、默认展示角色
  - 页面访问判定：看路由权限
  - 按钮与接口操作判定：看动作权限
  - 数据访问判定：统一按“仅自己 / 全部”范围执行
  - `用户表`、`权限表`：仅 `admin/root` 可见可配

## 已确认设计基线

- 设计第 1 段已确认：P1 采用封闭式接入模型，初始化单一团队与默认 `root` 账号；用户可绑定多个角色，权限点只挂角色；后端按角色权限点并集鉴权，前端按当前选中角色做入口与页面展示裁剪。
- 设计第 2 段已确认：系统以一次性、幂等初始化为起点，创建引导型 `Team Workspace`、默认 `root` 账号，以及 `root/admin/manager` 内置角色模板；成员生命周期仅走后台创建与禁用链路，不走注册、邀请与物理删除；其中 `root` 为应用级最高权限，`admin/manager` 为当前空间角色模板，为后续多空间或子租户预留分层扩展能力。
- 设计第 3 段已确认：权限判定采用“先应用级、后空间级”的双层模型；后端先判断当前用户是否为应用级 `root`，若否则按当前空间内用户角色权限点并集鉴权，并在数据层继续区分“自己的数据 / 全部数据”；前端只负责视角与入口裁剪，不承担安全边界，其中 `root` 进入全局管理视角，非 `root` 用户按当前选中角色切换界面视角。
- 设计第 4 段已确认：P1 权限矩阵按 6 组边界一次性收敛，即应用级权限、成员管理权限、团队配置权限、角色与权限配置权限、业务资源权限、页面与数据范围判定；其中 `root` 独占应用级治理，`admin` 管当前空间成员与配置，`manager` 默认只管自己业务资源，普通用户只保留个人资料、密码与默认展示角色的自助能力。
- 设计第 5 段已确认：普通成员忘记密码时不提供自助找回，而是由具备成员管理权限的 `admin/root` 在后台直接重置新密码；后台重置密码后不强制首次登录改密；该约束只作用于后台重置场景，不改变“新建账号首次登录不强制改密”的既有基线。
- 设计第 6 段已确认：P1 团队基础配置采用最小边界，仅包含团队名称、团队 Logo/图标、团队简介；默认语言、时区等展示偏好不放入团队配置，而继续视为个人设置或后续系统配置议题。
