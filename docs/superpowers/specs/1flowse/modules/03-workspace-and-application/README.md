## 本模块范围

- 工作台首页
- 应用创建、浏览、搜索
- 应用作为交付容器的边界
- 应用内导航骨架

## 已有基线

- `Application` 是用户直接管理的交付单元。
- 应用不作为顶层导航类目，而归属于工作台。
- 顶层导航固定为：`工作台`、`团队`、`后台设置`。
- 工作台负责创建应用、浏览应用、搜索应用、查看最近访问和发布状态概览。
- 进入应用后采用左右结构，左侧为功能导航，右侧为主工作区。

## 本轮用户输入（2026-04-10，03 讨论）

- 工作台首页直接展示应用列表，顶部仅放少量统计信息。
- 应用列表采用传统表格/列表形态，使用 Ant Design 组件，后端分页查询。
- 工作台列表最小字段为：`应用名称`、`简介`、`owner`、`最近更新时间`、`最近发布状态`、`最近访问时间`、`标签/分类`。
- 工作台筛选与排序最小集合为：按应用名搜索、按 `owner` 筛选、按最近更新时间排序、按发布状态筛选、只看“我的应用”。
- 创建应用最小表单为：必填 `应用名称`；可选 `应用简介`、`图标`、`分类/标签`；图标先直接使用 Ant Design 图标。
- 创建应用即初始化默认 Flow，最小骨架为 `开始（输入） -> LLM -> 结束（输出）` 三节点；其他能力沿默认值，不额外扩展初始化范围。
- 新建应用后的首页落点先进入“应用概览页”。
- 应用内左侧导航最终栏目与顺序参考 Dify ChatFlow，先收敛为：`编排`、`应用 API`、`调用日志`、`监控报表`。
- 应用设置边界至少覆盖：应用名称/简介/图标、应用删除、应用复制、协作者管理；发布按钮放在画布页；支持查看发布历史并恢复节点。
- 生命周期动作明确为：支持删除、支持复制、支持硬删除、不支持归档。
- P1 正式支持应用协作者；协作者为应用级模型，权限可等同 `owner`；`owner` 被禁用后不设计额外接管流程。
- 顶部统计收敛为：`应用总数`、`我的应用`、`草稿数`、`已发布数`。
- `owner` 字段定义为“主负责人”，默认创建人为 `owner`，后续允许转交。
- P1 仅保留一种删除语义：`删除 = 硬删除`，二次确认后不可恢复。
- `分类/标签` 建模收敛为“仅多标签”，不单独设置分类字段。
- 应用概览页最小内容收敛为：`基本信息 + 发布状态`。
- 应用信息架构方案采用 `方案 B`：应用根路由先落概览页，左侧导航仅保留 `编排 / 应用 API / 调用日志 / 监控报表` 四栏，设置/复制/删除放到应用头部操作区。
- 删除应用时直接清理对应关联数据，不保留删除后的历史统计或恢复链路；“恢复节点”仅来自发布历史，不来自删除后恢复。
- 工作台顶部统计先仅作展示，不联动列表筛选。
- 应用概览页保留单一主入口 `进入编排`，不额外提供 `查看 API / 查看日志` 快捷入口。
