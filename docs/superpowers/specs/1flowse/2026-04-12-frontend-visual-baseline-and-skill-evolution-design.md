## 1. 文档目标

本文档用于沉淀本轮已确认的三类共识：

1. `frontend-development` 后续应如何吸收默认视觉基线。
2. `Editor UI` 应如何作为前端视觉子规范落位。
3. `backend-development` 与后续 `QA` skill 应如何继续演进，而不是长期停留在散落 spec 中。

本轮目标不是立刻冻结最终视觉 token，也不是马上改写 skill，而是先把已确认方向写清楚，作为后续迭代 UI 与 skill 的共同依据。

## 2. 当前已确认现状

### 2.1 视觉来源

- 当前默认视觉来源先直接采用 [docs/draft/DESIGN.md](../../../draft/DESIGN.md)。
- 本轮不先抽象、不先改写母文档，先通过 demo 看真实效果，再决定保留与收敛内容。
- 该来源当前更像视觉母本，而不是已正式生效的项目规范。

### 2.2 视觉约束等级

- 1Flowse 前端视觉当前定位为：`默认基线`，不是“任何页面都绝不允许偏离”的硬锁死规范。
- 常规控制台页面默认遵守这套基线。
- 特殊模块允许受控偏离，但不能破坏整体产品统一感，也不能各自发展成独立产品语言。

### 2.3 壳层与画布关系

- 控制台壳层继续采用 `Ant Design` 作为成熟组件基础设施。
- 画布内部继续采用 `Editor UI` 作为薄自定义组件层。
- 两者不是两个视觉系统，而是同一系统下的两种表达层。
- `Editor UI` 需要比壳层更“紧”和“准”，但仍共享同一套色彩角色、字体节奏、圆角逻辑、阴影哲学与状态表达。

### 2.4 当前验证方式

- 本轮已先做静态 demo，路径为 [tmp/demo/index.html](../../../../tmp/demo/index.html)。
- demo 用于验证方向，不代表规范已定稿。
- 后续是否吸收进 skill，应以 demo 反馈和二轮视觉讨论结果为准。

## 3. 当前推荐方向

## 3.1 总体方向

1Flowse 前端应形成“一套默认视觉系统，两个表达层”的结构：

- `Shell Layer`
  - 面向控制台壳层、列表、表单、详情、抽屉、空态、配置页。
  - 以 `Ant Design` 为基础设施，但不直接裸用默认外观。
  - 通过 token 覆盖和少量壳组件，让整体风格回收到 1Flowse 自己的产品语气。
- `Editor UI Layer`
  - 面向画布、节点、端口、连线、工具栏、Inspector、局部运行状态。
  - 保持更高信息密度、更明确状态表达、更轻装饰。
  - 复杂字段、弹窗等交互仍可在自有壳组件内部适度复用 `Ant Design`。

## 3.2 默认视觉基线

后续默认视觉基线应优先吸收 `DESIGN.md` 中适合工具型控制台的部分，而不是照搬其更偏展示型或品牌宣传型的表达。

当前确认可优先吸收的方向：

- 白底或浅底的高对比工作区
- 近黑文本而非灰弱文本
- 明确、稳定的主色
- `4px-8px` 为主的锐利圆角体系
- 克制但有层次的阴影
- 工具型排版，而非营销型排版
- 小号 uppercase 标签用于层级提示
- 少量但语义明确的辅助状态色

当前明确不建议直接照搬的方向：

- 过强的营销页式视觉情绪
- 为了设计感牺牲信息密度
- 与产品状态语义无关的装饰性颜色使用
- 画布与壳层完全不同步的品牌表达

## 3.3 Editor UI 子规范方向

`Editor UI` 不应只是“画布专用随手组件集合”，而应有清晰的视觉子规范。

子规范当前方向为：

- 比壳层更高信息密度
- 比壳层更轻装饰
- 比壳层更强调 selected / waiting / failed / running 等状态差异
- 比壳层更强调节点、端口、连线和 inspector 的结构层级
- 但仍保持以下统一：
  - 主次色角色统一
  - 文字层级统一
  - 圆角和边框哲学统一
  - 阴影逻辑统一
  - hover / focus / selected 的反馈逻辑统一

## 4. 这样推进的后果

## 4.1 正向后果

- AI 后续开发前端时，不再每次重新发明 UI 风格。
- `Ant shell + Editor UI` 可以逐步收敛成一个产品，而不是后台和画布各长各的。
- 主 skill 保持轻量，reference 承载具体规则，能降低腐化速度。
- 后续新会话在低上下文条件下，也更容易复用一致的判断标准。

## 4.2 风险与代价

- 如果过早把 demo 中的所有细节都硬写进 skill，skill 会迅速过时。
- 如果只写抽象口号，不写半具体规则，默认基线仍然无法真正约束 AI。
- 如果不单独给 `Editor UI` 落子规范，画布区后续很容易重新长出独立审美。
- 如果后端稳定架构约束继续只散落在 spec 中，后续实现时仍会反复回查甚至漂移。
- 如果 QA 仍然只依赖当前开发会话上下文，AI 很容易对自己的产物过分宽容。

## 5. 我的建议

## 5.1 frontend-development 的落位建议

`frontend-development/SKILL.md` 后续应补入“半具体层”视觉规则，而不是停留在纯抽象原则。

建议保留在 `SKILL.md` 的内容：

- 默认视觉基线存在，普通控制台页面默认遵守
- 壳层优先复用 `Ant Design`，但不要直接裸用默认视觉
- 不新增新的主组件库与主样式框架
- 新页面默认服从已有页面语法，不随手新增一套视觉语法
- `Editor UI` 属于同一系统，但应遵守更紧凑的子规范
- 若页面视觉方案明显偏离默认系统，应先问人

## 5.2 frontend-development 的 reference 建议

建议至少新增两份 reference：

1. `references/visual-baseline.md`
   - 壳层默认视觉基线
   - 色彩角色
   - 字体气质
   - 圆角 / 边框 / 阴影
   - 列表 / 表单 / 抽屉 / 详情 / 空态的共同语法
   - 常见反例

2. `references/editor-ui-visual-spec.md`
   - `EditorSurface`
   - `EditorToolbar`
   - `NodeCard`
   - `NodePort`
   - `Inspector`
   - 选中态、等待态、失败态、运行态的表达规则
   - 为什么 `Editor UI` 要更紧、更准、更少装饰

## 5.3 backend-development 的演进建议

后端最底层最核心架构相对稳定，不应长期只作为项目 spec 零散存在。

建议后续把以下内容逐步吸收进 `backend-development`：

- 哪些边界属于稳定核心，不允许被随意改写
- 插件边界为何属于高风险变更
- 状态入口为何必须唯一
- `domain / adapter / storage / publish / runtime` 的职责边界
- 哪些改动必须先问人，而不是让 AI 自行决策

当前判断：

- 插件作者指南或独立 `plugin-development` skill 不是此刻的首要落点。
- 更优先的是让 `backend-development` 能承载并守住稳定架构与契约边界。

## 5.4 QA skill 的演进建议

建议后续新增一个独立 QA skill，定位不是“开发顺手跑一下”，而是“可在新会话里低干扰评估整个项目质量”的评估器。

建议边界如下：

- 默认只出问题报告，不直接改代码
- 支持两种模式：
  - `task mode`
    - 只验证当前任务相关改动
    - 开发前应先写测试用例与验收场景
    - 开发后针对这些用例做回归测试
  - `project evaluation mode`
    - 只有用户明确要求“评估项目”时才启动全量模式
    - 检查 UI 一致性、浏览器截图、交互流、响应式、API 契约、状态与数据一致性、架构边界、测试缺口

这样做的原因是：

- 单一开发会话里的 AI 会受到上下文和自身实现路径影响，容易对成果过分宽容。
- 新会话的独立评估 skill 更适合承担“挑刺”和“找缺口”的职责。

## 6. 后续顺序建议

本轮之后建议按以下顺序继续：

1. 继续围绕 demo 讨论 UI 问题与视觉偏差。
2. 根据反馈决定默认视觉基线哪些保留、哪些修正。
3. 再更新 `frontend-development` 的 `SKILL.md` 与 references。
4. 随后补强 `backend-development` 中的稳定架构约束。
5. 最后再单独设计并实现 QA skill。

## 7. 当前结论摘要

- 当前先把 [docs/draft/DESIGN.md](../../../draft/DESIGN.md) 当作视觉母本，通过 demo 验证，而不是立刻抽象成最终规范。
- 前端视觉采用“默认基线”而不是“绝对硬约束”。
- 壳层与 `Editor UI` 属于同一产品系统，前者偏稳，后者偏紧，但不允许各自发展成完全不同的视觉语言。
- `frontend-development` 后续应补入半具体层视觉规则，并新增壳层与 `Editor UI` 两份 reference。
- 后端稳定架构约束应逐步进入 `backend-development`，而不是长期散落在 spec 中。
- QA 应作为独立 skill 演进，支持任务级回归与项目级全量评估两种模式。
