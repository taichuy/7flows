# 2026-03-14 开发收尾测试基线偏好

## 背景

用户在本轮任务中明确要求，把“开发完成后必须先做充分测试；测试不通过就继续修复代码或更新测试脚本；直到零警告、零错误再进入下一轮开发”沉淀为长期有效偏好，而不是停留在一次性口头提醒。

## 目标

- 将测试收尾标准记录为稳定偏好，供后续所有开发轮次复用。
- 把该偏好同步到仓库级协作规则，避免只存在于 `user-preferences` 中而执行时被遗漏。
- 在当前事实索引中明确交付基线，确保后续开发优先级和收尾动作保持一致。

## 决策与落地

- 在 `docs/dev/user-preferences.md` 新增“开发收尾必须充分测试并清零告警”偏好条目。
- 在 `AGENTS.md` 的“每次完成一轮开发后，额外要求”中补充测试收尾规则，要求充分测试、失败继续修复、默认以零警告零错误交付。
- 在 `docs/dev/runtime-foundation.md` 的“当前判断”中补充当前开发收尾基线，作为后续任务的默认事实约束。

## 影响范围

- 整个仓库后续的开发收尾方式、测试执行顺序与缺陷修复节奏。
- 最终汇报中对验证结果、剩余阻塞项和未达标原因的表达方式。
- 未来若引入更系统化的自动化测试与 CI，也需要沿用这条“验证不过关不算完成”的基线。

## 验证方式

- 通过 `git diff --check` 验证本轮文档修改没有引入格式层面的明显问题。
- 回读 `AGENTS.md`、`docs/dev/user-preferences.md`、`docs/dev/runtime-foundation.md`，确认三处表述一致、语义闭环。

## 未决问题 / 下一步

- 当前这条规则已落为协作基线，但仓库层面的“充分测试”仍需随业务推进逐步细化为更稳定的自动化检查组合。
- 后续可以结合 `api` 的 `pytest`、`ruff` 与 `web` 的 `lint`，继续沉淀成更明确的默认验证矩阵。
