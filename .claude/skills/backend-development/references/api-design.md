# AI-Friendly API Design

## Rules

| 规则 | 为什么 | 该怎么做 |
| --- | --- | --- |
| 参数名统一且清晰 | 降低构造错误率 | 使用稳定命名，避免同义字段并存 |
| 请求字段少 | 降低一次性拼装成本 | 一次只做一个动作 |
| 嵌套浅 | 降低模型构造深层 JSON 的错误率 | 优先拆成多次查询或多步动作 |
| 单次语义单一 | 便于理解和重试 | 一个接口只表达一个业务动作 |

## Split It Instead

遇到以下情况，不要继续堆字段：

- URL 参数长到难读
- 请求体嵌套多层
- 一个动作同时包含查询、校验、提交、发布
- 调用方必须知道太多内部结构才能发请求

此时优先：

- 拆成预查询和正式动作
- 拆成主动作和附属动作
- 在适配层做映射，不把调用负担压给调用方
