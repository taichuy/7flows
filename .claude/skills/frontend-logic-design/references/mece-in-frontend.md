# MECE 原则在前端设计中的应用

## 什么是 MECE

MECE（Mutually Exclusive, Collectively Exhaustive）由麦肯锡咨询顾问芭芭拉·明托在《金字塔原理》中提出，意为"相互独立，完全穷尽"。

- **互斥（ME）**：每个元素只属于一个分类，分类之间无重叠
- **穷尽（CE）**：所有元素都能归入某个分类，无遗漏

## 在前端设计中的核心应用

### 1. 选择正确的分类维度

MECE 的难点不在于"不重不漏"，而在于**选择正确的分类维度**。

**错误示范**：按 UI 容器分类
```
侧栏：任务详情
页面：任务管理
对话：AI 执行
```
→ 问题：如果新增一个"任务审批"功能，它应该放在侧栏还是页面？分类依据不清晰。

**正确示范**：按信息深度分类
```
L0 概览：摘要信息，直接可见
L1 聚焦：单条记录详情，点击触发
L2 管理：全量数据操作，导航触发
L3 执行：AI/复杂操作，行动按钮触发
```
→ 新增功能时，先判断它的信息深度，容器自然确定。

### 2. 导航结构的 MECE

一级导航的每个菜单项必须满足 MECE：
- 互斥：用户不会困惑"这个功能应该在菜单A还是菜单B"
- 穷尽：用户不会找不到某个功能的入口

**检验方法**：拿出系统的任意一个功能，问"它属于哪个导航项？"如果答案模糊或有多个，说明分类不互斥。

### 3. Tab/筛选器的 MECE

Tab 和筛选器是 MECE 最直接的应用场景：

```
✅ 好的 Tab 设计（MECE）：
  全部 | 待处理 | 进行中 | 已完成
  → 互斥：一个任务只能处于一种状态
  → 穷尽：所有状态都被覆盖

❌ 坏的 Tab 设计（非 MECE）：
  待处理 | 我的任务 | 紧急 | 已完成
  → 不互斥："我的任务"可能和"待处理"重叠
  → 维度混乱：状态/归属/优先级三个维度混在一起
```

### 4. 用户角色的 MECE

为不同角色设计差异化体验时，角色分类必须 MECE：

```
✅ MECE 角色分类：
  超级管理员 | 部门负责人 | 普通员工
  → 互斥：一个用户在同一上下文只有一个角色
  → 穷尽：所有用户都能归入某个角色

❌ 非 MECE 角色分类：
  管理者 | 执行者 | VIP 用户
  → 不互斥：管理者也可能是 VIP
```

## 实践方法：二分法兜底

当找不到完美的 MECE 分类时，使用**二分法**作为起点：

```
A / 非A
→ 再对 A 和非 A 分别细分
→ 层层递进，直到粒度足够
```

例如：用户是否已登录？→ 已登录用户是否有权限？→ 有权限用户的角色是什么？

## 注意事项

1. **MECE 是分类工具，不是目的**。如果严格 MECE 导致用户体验变差（比如为了互斥把相关信息拆得太碎），适当放松是合理的。
2. **允许数据重叠，不允许操作重叠**。同一个任务可以同时出现在"今日工作"和"待办事项"卡片中（呈现角度不同），但"删除任务"这个操作只应该在一个地方可以执行。
3. **分类维度要与用户心智模型一致**。按公司内部组织结构分类往往不如按用户使用场景分类有效。
