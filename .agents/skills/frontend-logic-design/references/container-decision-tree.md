# UI 容器决策树：Modal vs Drawer vs Page vs Inline

## 核心原则

UI 容器的选择不是审美问题，而是**信息架构问题**。选择依据是：
1. 用户是否需要保持当前上下文
2. 操作的复杂度和持续时间
3. 是否需要参考/对比背景信息

## 决策树

```
用户需要保持当前页面的上下文吗？
│
├─ 是（用户操作时需要看到/参考当前页面内容）
│   │
│   ├─ 操作内容量大吗？（超过一屏的信息/多个表单字段）
│   │   ├─ 是 → 【Drawer 侧栏抽屉】
│   │   │   适用：查看记录详情、编辑表单、子任务列表
│   │   │   优点：保持上下文，空间充足
│   │   │   注意：移动端需要降级为全屏
│   │   │
│   │   └─ 否 → 操作是否需要用户确认/决策？
│   │       ├─ 是 → 【Modal 弹窗】
│   │       │   适用：删除确认、审批确认、简单选择
│   │       │   优点：聚焦注意力，防止误操作
│   │       │   注意：不要用于复杂表单
│   │       │
│   │       └─ 否 → 【Inline 就地操作】
│   │           适用：状态切换、一键审批、标记已读
│   │           优点：零打断，最高效
│   │           注意：需要明确的反馈（toast/状态变化）
│   │
│   └─ （小技巧：如果犹豫 Drawer 还是 Modal，选 Drawer）
│
└─ 否（操作独立于当前页面）
    │
    ├─ 操作复杂度高吗？（多步骤/全量数据/批量操作/需要全屏空间）
    │   ├─ 是 → 【Page 独立页面】
    │   │   适用：全量列表管理、多步骤向导、搜索结果、报表
    │   │   优点：最大操作空间，支持 URL 直达和浏览器后退
    │   │   注意：需要清晰的返回路径（面包屑/返回按钮）
    │   │
    │   └─ 否 → 【Modal 弹窗】或 【Drawer 侧栏】
    │       根据内容量选择
    │
    └─ （小技巧：如果犹豫 Page 还是 Drawer，看操作是否需要 URL 可分享）
```

## 各容器的适用场景速查

### Inline（就地操作）
```
✅ 适用                          ❌ 不适用
- 状态切换 toggle               - 需要输入文本的操作
- 审批通过/驳回按钮             - 不可逆操作（需确认）
- 标记已读/未读                 - 需要展示额外信息的操作
- 收藏/取消收藏
```

### Modal（弹窗）
```
✅ 适用                          ❌ 不适用
- 删除/取消等不可逆确认          - 复杂表单（>5个字段）
- 简单选择（1-3个选项）          - 需要参考背景的操作
- 安全验证/权限确认              - 多步骤流程
- 短消息/通知展示                - 错误信息展示
                                 - 首次引导/onboarding
```

### Drawer（侧栏抽屉）
```
✅ 适用                          ❌ 不适用
- 记录详情查看                   - 需要全屏对比的场景
- 中等复杂度的表单编辑            - 需要 URL 直达的内容
- 子列表/关联信息展示             - 移动端复杂交互
- 需要同时参考主列表的操作
```

### Page（独立页面）
```
✅ 适用                          ❌ 不适用
- 全量数据列表 + 搜索筛选         - 快速预览单条记录
- 多步骤向导/流程                 - 简单确认操作
- 复杂表单（>10个字段）           - 需要保持上下文的场景
- 报表/图表/数据分析
- 需要 URL 可分享/书签的内容
```

## 在看板场景中的典型应用

| 交互 | 容器选择 | 原因 |
|------|---------|------|
| 点击任务条目查看详情 | Drawer | 保持看板上下文，方便快速切换 |
| 审批通过/驳回 | Inline | 一键确定性操作，无需离开 |
| 查看全部任务 | Page | 全量数据+搜索筛选，需要完整空间 |
| AI 执行确认 | Modal → Page | 先 Modal 确认意图，再跳转对话页面执行 |
| 创建新项目 | Modal | 字段少（名称+链接），不需要独立页面 |
| 刷新数据 | Inline | 按钮触发，loading 状态，就地完成 |

## Smashing Magazine 的四步决策法

来自 Ryan Neufeld 的决策框架，Smashing Magazine 总结：

1. **上下文检查**：用户操作时是否需要看到底层页面？
2. **复杂度检查**：操作是简单聚焦还是多步骤复杂流程？
3. **参考检查**：用户是否需要对比/参考其他数据？
4. **容器选择**：基于以上三点选择 Modal / Drawer / Page

> "Whenever possible, avoid blocking the entire UI. If you want to boost users' efficiency and speed, avoid modals at all costs. Use them to slow users down, to bundle their attention, to prevent mistakes."
> — Vitaly Friedman, Smashing Magazine
