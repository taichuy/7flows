# 7Flows 自治开发工程判断尺

## 1. 第一原则

自治开发的意义不是给 AI 更大的自由，而是让 AI 在每一轮都按同一套工程观判断：

- 现在项目处于什么阶段
- 当前最主要的问题是什么
- 哪个改动最能推进主链闭环
- 什么属于必要结构增强，什么只是局部优化

如果这些判断不存在，AI 就只是在“自动做事”，而不是“按工程观自治开发”。

## 1.1 提示词设计原则

一个好的自治开发提示词，不是把所有原则和所有细节都塞进去，而是做到：

- 少量高质量原则
- 明确优先级
- 明确禁止项
- 固定复盘格式

提示词一旦过满，常见后果是：

- AI 抓不住真正高优先级
- AI 机械套原则，开始过度设计
- 每轮都花很多 token 重复世界观，却没有稳定推进主链

因此：

- “好 / 坏标准”要保留
- 但长例子、过多术语、重复解释应下沉到 reference，而不是都放进每次会话开头的提示词

## 2. 好项目的标准

一个“好项目”，至少表现为：

- 主链路清晰，而且在一轮一轮被真正补齐
- 用户层、AI 与人协作层、AI 治理层同步向产品目标收敛
- 关键事实有唯一来源，状态、动作和追溯入口不分裂
- 架构能继续承接插件扩展、兼容演进、可靠运行和安全治理
- 文档、代码结构和运行事实能支撑后续持续迭代，而不是只服务当前一次提交

对 7Flows 来说，要特别警惕“外部切口”和“内部内核”混线：

- 对外可以 OpenClaw-first
- 对内仍必须坚持 `7Flows IR`、统一 runtime、统一事件流、统一发布映射

## 3. 好代码的标准

一个“好改动”，至少满足：

- 先正确，再谈优雅
- 关键业务规则与数据库细节、第三方协议、展示文案解耦
- 对外契约稳定，内部实现允许演进
- 改动范围与问题范围匹配，不为了一个小问题撬动太多无关模块
- 读者能定位边界、依赖和意图，后续人类与 AI 都容易搜索、理解和修改

如果一个改动只是：

- 改名字更顺眼
- 文件更整齐
- 组件更碎
- 抽象层更多

但没有让主链更完整、边界更稳定、验证更可靠，那它不算“更好”。

## 4. 好架构的标准

一个“好架构”不是更抽象，而是：

- 稳定核心承载业务规则，变化实现被收敛在边界和适配层
- 业务层决定“该不该做”，基础设施层负责“怎么做”
- 外部变化不会直接污染内部事实模型
- 关键模型、关键状态、关键动作有清晰唯一入口
- 状态名之外，还有状态流转规则和动作约束

对 7Flows，要特别看这几个边界有没有被守住：

- `7Flows IR` 是否仍是内部事实中心
- workflow executor 是否仍是唯一主控
- `runs / node_runs / run_events` 是否仍是运行事实来源
- `execution class` 和 `sensitivity_level` 是否仍分离建模
- compat adapter 和 sandbox backend 是否仍职责分离

## 5. 坏信号清单

以下情况默认视为风险，而不是“看起来也做了很多工作”：

- 长时间停在局部修补、样式整理、命名清洗、抽象洁癖
- 架构复杂度上升，但没有提升扩展性、兼容性、可靠性、稳定性或安全性
- 一个小改动要同步改很多层，说明变更传播过大
- 业务规则、存储细节、第三方协议、展示拼装混在一起
- 过度拆分导致理解路径更长，或者过度聚合导致热点文件不断吸职责
- 每轮都重新做全局方向判断，导致主线迟迟不推进

## 6. 单轮候选任务评分

为每个候选任务按下面规则打分：

### 正向分

- `主链闭环价值`：0-3
  - 3 = 明显补齐用户层 / AI 协作层 / AI 治理层中的关键断点
  - 2 = 推进某个重要子链，但不是最核心断点
  - 1 = 有帮助，但偏间接
  - 0 = 主要是局部优化
- `阻塞消除价值`：0-3
  - 3 = 不做就明显挡住后续 `P0 / P1`
  - 2 = 会显著降低后续实现成本
  - 1 = 只是轻微改善
  - 0 = 不影响后续主线
- `共享边界价值`：0-2
  - 2 = 让多个入口复用同一事实、同一 helper、同一治理规则
  - 1 = 只改善局部一致性
  - 0 = 没有共享增益
- `可靠性 / 安全性 / 兼容性价值`：0-2
  - 2 = 明显提高诚实性、fail-closed、观测性或兼容稳定性
  - 1 = 有轻微提升
  - 0 = 基本无影响

### 负向分

- `局部优化倾向`：0-3
  - 3 = 基本是局部美化或局部清洁
  - 2 = 收益主要停留在局部
  - 1 = 有一定局部优化成分
  - 0 = 明显不是局部优化
- `实现风险 / 扩散成本`：0-2
  - 2 = 容易牵连很多无关模块或引入大回归风险
  - 1 = 有一定外溢成本
  - 0 = 风险可控
- `边界破坏风险`：0-2
  - 2 = 可能冲击既有架构硬边界
  - 1 = 有轻微边界漂移风险
  - 0 = 不会破坏边界

## 7. 选题公式

默认总分：

`(主链闭环 + 阻塞消除 + 共享边界 + 可靠性安全兼容) - (局部优化 + 实现风险 + 边界破坏)`

默认选择：

1. 总分最高的候选任务
2. 若总分接近，优先 `runtime-foundation.md` 里优先级更高的项
3. 若仍接近，优先可在当前一轮形成可验证闭环的项

## 8. 功能、架构、解耦的决策规则

### 功能优先于非阻塞重构

如果关键场景仍未闭环，而现有结构尚能承载下一步实现：

- 优先补功能闭环
- 不优先做“看起来更优雅”的结构整理

### 只有阻塞型架构问题才优先处理

若满足以下任一条件，可优先做架构增强：

- 当前结构已经让 `P0 / P1` 无法继续诚实落地
- 同类变更连续跨 2 个以上模块重复传播
- 运行事实、状态入口或治理边界已经出现明显分裂
- 不先修边界，就会继续堆出更多临时补丁

### 文件拆分判断

文件超出体量阈值只是预警，不是自动拆分命令。

优先同时看：

- 是否存在多种变化原因混在一起
- 是否业务规则、流程编排、存储细节、第三方集成或展示拼装混杂
- 是否改一次就牵连很多区域
- 是否已经影响后续维护、扩展、检索和追踪
- 拆分后是否同时对人类与 AI 更友好

只有行数大，但边界清晰、变化原因单一时，不要为拆而拆。

## 9. 单轮执行协议

每一轮默认遵守：

1. 先判断阶段问题
2. 再列候选任务
3. 用评分规则选一个主主题
4. 最多附带一个服务于主主题的结构调整
5. 做实际改动
6. 验证
7. 更新共享事实入口；如当前开发者需要个人连续性，再写本地留痕
8. 给出下一步优先级

如果当前轮次只是：

- 提示词优化
- 协作规则设计
- 方案比较
- 事实评估

可以不改业务代码，但仍应产出可复用的协作资产，而不是只留对话结论。

## 10. 对外 prompt 骨架

下面这段比“请你自主开发项目”更接近可执行版本：

```text
你不是为了“自主”而自主，而是要按 7Flows 已确认的工程观持续自治开发项目。

每轮开始前：
1. 先读取 AGENTS.md、docs/dev/user-preferences.md、docs/dev/runtime-foundation.md，并检查 git status。
2. 如果任务涉及产品边界、sandbox、安全、插件兼容或发布协议，再补读产品/技术/策略文档；如果当前本地开发者需要衔接自己的过程记忆，可再查看 docs/.private/history/。
3. 先判断当前项目的主要阶段问题，再从 runtime-foundation 的最高优先级未完成项中选择本轮主题；若代码事实证明另一个阻塞更紧急，可以改选，但必须说明依据。

你的判断标准不是“哪里都能优化”，而是：
- 好项目：主链路逐步闭环，用户层 / AI与人协作层 / AI治理层持续收敛
- 好代码：先正确，再易懂、易改、边界清晰、可测试
- 好架构：稳定核心承载业务规则，变化实现收敛在边界和适配层
- 坏信号：长时间停在局部修补、样式整理、抽象洁癖或无收益复杂化

优先级规则：
- 优先推进全局场景完整度和主链闭环
- 优先解决阻碍功能推进、扩展、可靠运行的问题
- 只有当架构问题已经成为推进阻塞时，才优先做架构增强
- 只有职责混杂、变更传播过大、明显影响维护和扩展时，才做文件解耦

单轮只推进一个主主题，最多附带一个服务于主主题的结构调整。不要为了“看起来更完整”而同时铺太多方向。

每轮结束后必须回答：
1. 当前项目的主要阶段问题
2. 为什么选择本轮任务
3. 本轮做了什么实际改动和验证
4. 当前愿景目标完整度：哪些层已经有骨架、哪些层已有最小闭环、哪些层仍是全局缺口
5. 这次改动属于什么类型
6. 它帮助了哪些方面
7. 当前还缺哪些关键环节
8. 本轮是否偏离主链，为什么
```

如果需要真正用于“每次新会话开头”的版本，优先使用同目录的 `session-bootstrap-prompt.md`，而不是继续向这段骨架里追加细节。

## 11. 愿景目标完整度视图

每轮自治开发汇报都应带一份“愿景目标完整度”视图，用来提醒 AI 和人不要只盯着当前任务。

推荐至少覆盖：

- 用户层完整度
- AI 与人协作层完整度
- AI 治理层完整度
- 运行基础 / 发布 / 兼容层完整度

表达方式建议：

- 不强求伪精确百分比
- 优先用“未成形 / 有稳定骨架 / 已有最小闭环 / 进入细化完善”这类阶段判断
- 同时写清楚当前最大的全局缺口是什么，而不是只报“完成了多少”

一个合格的完整度视图，至少要回答：

1. 整体愿景是什么
2. 当前已到哪一阶段
3. 本轮让哪一部分更完整
4. 距离下一阶段还差哪些关键环节
