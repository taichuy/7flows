# 7Flows 自治开发

## 何时使用

当任务涉及以下任一场景时使用：

- 用户明确要求“持续自治开发”“自主推进项目”“自己判断优先级并实现”
- 需要在多个候选方向里判断本轮最值得做的主题，而不是只执行一个明确的小改动
- 需要判断当前是该补功能闭环、补运行基础、消除阻塞型架构问题，还是只需要轻量整理

不要用于：

- 用户已经明确指定了单一、局部且边界清楚的任务
- 纯 brainstorm、纯观点讨论，尚未进入可执行落地阶段

## 先建立哪些事实

至少复核：

- `AGENTS.md`
- `docs/dev/user-preferences.md`
- `docs/dev/runtime-foundation.md`
- `git status --short`
- 本轮最可能触达的模块

若任务涉及产品切口、版本边界、插件兼容、sandbox、安全或运行时模型，再补读：

- `docs/product-design.md`
- `docs/technical-design-supplement.md`
- `docs/open-source-commercial-strategy.md`
- 相关 `docs/history/*.md`

若任务进入具体领域实现，再组合对应领域 skill：

- 后端：`backend-code-review` / `backend-testing`
- 前端：`frontend-code-review` / `frontend-testing` / `component-refactoring`
- 收尾：`development-closure`

## 核心原则

- 自治不是“自由发挥”，而是按 7Flows 已确认的工程观和当前事实，选择本轮最高杠杆任务。
- 先判断阶段问题，再决定做什么；不要先挑自己擅长或看起来舒服的任务。
- 默认从 `docs/dev/runtime-foundation.md` 的最高优先级未完成项中选题，除非代码事实证明有更紧急的阻塞。
- 单轮只推进一个主主题，最多附带一个为主主题服务的结构调整。
- 架构不是目的；只有当架构问题已经阻塞当前 `P0/P1` 主线时，才优先处理架构。
- 文件拆分不是目的；只有当职责混杂、变更传播过大、检索维护困难，或已经同时影响人类与 AI 修改效率时，才做解耦。

详细的“好项目 / 好代码 / 好架构 / 坏信号 / 单轮评分规则”见 `references/engineering-rubric.md`。

## 单轮决策流程

### 1. 先判断当前阶段问题

至少按三层复核：

- 用户层：主链路是否更完整，作者 / operator / 调用方是否更容易完成关键动作
- AI 与人协作层：运行事实、排障入口、续跑与解释是否更统一
- AI 治理层：敏感访问、执行隔离、审计、发布边界是否更诚实

并在开始实现前先形成一份“愿景目标完整度”快照，至少回答：

- 完整目标中的哪些关键层已经有稳定骨架
- 哪些层已经形成最小闭环
- 哪些层仍是当前全局最大缺口
- 本轮若成功，会推动哪一层完整度前进

### 2. 再生成候选任务

候选任务默认来自：

- `runtime-foundation.md` 当前 `P0/P1`
- 当前代码里的真实阻塞热点
- 与本轮主题直接相关的文档或 skill 漂移

不要把“可以做”自动当成“现在最该做”。

### 3. 用工程 rubric 选题

- 先看它是否推进主链闭环
- 再看它是否消除后续开发阻塞
- 再看它是否增强共享能力边界、可靠性或安全性
- 最后才看美观度、命名整洁度或局部一致性

若一个候选任务主要只是：

- 样式整理
- 表层抽象
- 无明确业务收益的大重构
- 与当前 `P0/P1` 主线无关的局部修补

默认不优先。

## 执行约束

- 先复用现有结构、helper、service 边界和 skill，不从零再造流程。
- 不重开已经在产品/技术基线中定下的边界，除非代码事实已经明显冲突。
- 若工作区已有无关脏改动，默认避开；若必须进入同一区域，先理解现状再继续，不覆盖未知变更。
- 改动应尽量形成“下一轮可继续接”的稳定中间态，而不是只留下半成品抽象。

## 收尾要求

- 如果本轮形成 durable change，收尾时必须组合 `development-closure`
- 同步补齐需要的 `docs/history/`、`docs/dev/runtime-foundation.md`、`docs/dev/user-preferences.md`
- 给出按优先级排序的下一步建议
- 最终汇报必须显式包含“愿景目标完整度”视图，帮助后续判断当前整体阶段与剩余关键缺口

## 常见反模式

- 把“自治”理解成每轮都重新发明产品方向
- 长时间困在局部修补、样式整理或抽象洁癖里
- 用“先做架构”回避真实的主链闭环问题
- 没有依据当前代码事实和 `runtime-foundation`，只凭泛化经验选题
- 明明是多模块阻塞，却继续在单点上打补丁
