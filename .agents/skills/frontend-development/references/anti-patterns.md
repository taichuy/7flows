# Frontend Anti-Patterns

| 反模式 | 风险 | 替代方式 |
| --- | --- | --- |
| 把外部视觉样本直接当项目规范 | 当前产品语法被外部品牌风格覆盖 | 以 `docs/draft/DESIGN.md` 为准，外部样本只局部借鉴 |
| 页面 God Component | 一个页面改动牵动所有逻辑 | 按变化原因拆出局部状态、局部视图、协议适配 |
| 状态四处散落 | Bug 难追、行为不稳定 | 先定义状态归属，再决定放置层级 |
| 为单次使用强行抽象 | 抽象先于需求，后续更难改 | 保持内联，等第二个真实场景再抽 |
| 协议拼装和渲染混写 | 前端被接口细节污染 | 先在适配层做转换，再交给视图层 |
| 顺手造新交互语法 | 界面一致性被打碎 | 复用既有交互语法，必要时先找人确认 |
| 同类区域点击行为不一致 | 用户心智混乱 | 把问题上升到结构层处理，不做局部打补丁 |
| `overview` 页越界承接重内容 | 根路由失去“概览 + 单一入口”职责 | 回到固定 recipe，只保留头信息、状态、最近运行和 `进入编排` |
| 在 Shell / Canvas 间混用 L1 模型 | 同类对象的详情路径不可预测 | Shell 列表统一 `Drawer`，Canvas 对象统一 `Inspector` |
| 状态色复用为类型色或装饰色 | 运行态语义失真，跨页面不可比对 | 类型 badge 保持中性，状态只表达系统真相 |
| 把桌面画布硬塞进小屏 | 产生半可用界面，破坏首屏优先级 | 移动端做摘要态和明确引导 |
