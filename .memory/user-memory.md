# 用户记忆

## 用户画像

- 当前角色：产品负责人、创始人、项目全栈开发者。
- 技术背景：5 年开发经验，其中约 3 年后端、2 年全栈；熟悉 `Java`、`Node.js`、`Python`。
- 讨论深度：默认可直接使用工程、架构、产品交互术语；用户看不懂会追问。
- 关注重点：产品交互体验、架构边界、数据一致性、状态一致性、AI 协作效率。

## 沟通偏好

- 默认使用中文交流；设计文档默认中文，implementation plan 默认英文，git commit message 使用英文。
- 先给判断和核心原因，再补充必要细节；有明确技术倾向时直接亮明推荐。
- 不用客服腔、汇报腔和征询式收尾，避免“结论”“有以下几点需要说明”“如果你愿意”“整体是合理的”等套话。
- 控制层级深度，避免大段嵌套列表；同一观点只用一种清晰逻辑讲一遍。
- 架构和设计讨论中优先使用全称；不要把“数据建模定义”和“大语言模型 / 人工智能模型”都简称为“模型”。
- 涉及 UI / UX、流程、结构关系时，默认用 ASCII UI 或简图辅助说明。

## 决策方式

- 清晰任务直接推进；缺失信息不会实质改变结果时，用合理假设继续。
- 需要用户拍板的问题集中列出，不要多轮零散追问。
- 多选或模糊问题使用统一格式：现状、选择、后果、好处和风险、明确建议。
- 用户写明“直接开始无需询问”或同等含义时，同一任务链路连续推进；无法自行决策的事项写入 `.memory/todolist/`。

## 执行偏好

- 执行前先列计划再动手；进入实现后先整体完成，再根据验证结果和报错逐项修正。
- 当前项目按开发初期处理，优先完整一致性；默认不做兼容、回退逻辑和自动迁移，除非用户明确要求。
- 执行计划或需要独立 agent / 子 agent 协作时，同一时间只运行一个独立 agent（测试 agent 不受限）；每完成一个计划任务同步更新计划文档。
- 执行实现计划时不使用 `git worktree`，直接在当前仓库通过本地统一版本管理推进。
- 单个 implementation plan 超过 800 行时，拆成 `1+n` 结构：一个索引计划 + 多个执行计划；索引计划携带 `index`，子计划用 `01`、`02`、`03` 编号。
- 有落地文件变更且验证通过后，默认提交本次相关变更并仅 push 当前分支；纯分析、等待确认、未解决冲突或验证不足时不自动提交 / push。
- push 时除非用户明确要求合并推送、同步其他分支或指定分支，否则只推送当前分支。

## 验证偏好

- UI 调整和样式修复优先快速完成并给出可见效果；验证默认跑目标测试或最小相关测试，不主动跑前端全量测试。
- 前端页面的打开、检查、截图和交互复现默认使用 Playwright；不主动使用 Chrome 浏览器 MCP。
- 后端 Rust 数据一致性、状态一致性等重型门禁默认放到 GitHub Actions；除非用户明确要求，不在本地运行这类重型后端门禁。
- 工作结束总结中，主要代码文件需要说明：文件负责什么、本轮具体做了什么。

## 设计与界面偏好

- 工具型页面的顶部状态行、摘要行和轻量操作区保持紧凑透明；不要默认做成高白色卡片。
- 文件管理体验倾向表格 + 抽屉。
- Settings 页面倾向共享 surface，避免每个区块割裂成独立卡片。
- 节点动态字段倾向浮层对话框；变量列表倾向单行拖拽；变量选择浮层应逃逸被裁剪容器。

## 复盘偏好

- 做“昨天 / 今天热点修改、反复改动、AI 编程效率优化”类复盘时，重点是抽象 AI 下次如何避免同类错误，并落到 skills、AGENTS、质量脚本或代码环境规则。
