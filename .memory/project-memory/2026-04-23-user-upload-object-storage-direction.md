# 用户上传文件与多模态产物改用独立对象存储

## 时间

`2026-04-23 20`

## 谁在做什么

- 用户正在规划后续“对话上传文件、生成图片、多模态附件”能力。
- AI 后续需要按该方向设计独立的文件存储边界与上传链路。

## 为什么这样做

- 当前仓库里真正落盘的文件主要是插件产物，目录与生命周期都按安装包管理设计，不适合承接用户业务文件。
- 多模态场景会出现用户上传文件、模型中转文件和模型生成文件，这类对象需要独立的权限、生命周期和引用关系。
- 用户希望参考 `nocobase` 的分离方式，把“文件记录”和“存储器配置”拆开，避免业务直接依赖某一种具体存储实现。

## 为什么要做

- 让 API 能稳定接收并转发大文件给外部模型。
- 让生成图片等多模态产物有统一的持久化位置和元数据记录方式。
- 避免把插件安装目录和业务附件目录混在一起，导致权限、清理和审计边界失控。

## 截止日期

- 未指定

## 决策背后动机

- 用户已在 Docker 环境中部署 `rustFS`，准备用它承接业务文件对象存储。
- 后续用户上传文件、模型输入附件、生成图片等多模态产物应走独立对象存储，不复用 `api/plugins` 下的插件产物目录。
- 主仓官方持久化主存储仍以 PostgreSQL 为准；文件二进制走对象存储，数据库只保存元数据、引用关系和状态。
- 文件管理第一版按统一管理处理，用户上传文件与模型生成产物都进入统一文件体系。
- 需要提供“文件记录表 + 存储器表/文件管理器”分离架构，文件记录通过 `storage_id` 之类的中间层绑定到具体存储器。
- 默认存储为本地文件系统；`rustFS` 不默认启用，用户按需在后台启用和配置。
- 官方首批支持矩阵固定为“本地文件 + rustFS”；`S3`、阿里对象存储、腾讯云对象存储等后续按扩展或插件方式接入。
- 需要支持新建文件管理表，并让文件管理表绑定指定文件管理器，而不是整套业务只绑定单一全局存储。
- 文件体系第一版采用分层作用域：存储器由 `root/system` 统一管理；`root` 空间拥有默认文件表；`workspace` 可以新建自己的文件表，但不能修改存储器配置。
- 文件表与存储器的绑定关系只允许 `root` 修改，普通 `workspace` 只能选择和消费可见文件表。
- 文件表方案已选定为“真实多文件表 + 存储器绑定”，不是单表逻辑分组。
- 系统内建默认 `attachment/files` 文件表模板；新建文件表第一版直接复用这套固定字段模板，不支持额外自定义扩展字段。
- 文件表权限不单独发明新模型，直接复用动态建表运行态数据权限体系，按 CRUD 与 `own/all` 范围控制。
- 文件表改绑存储器第一版只影响之后的新上传；旧文件继续保留原 `storage_id`，不做迁移任务。
