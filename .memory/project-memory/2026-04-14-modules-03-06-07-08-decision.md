## 为什么要做

- `03` 是后续 `04 agentFlow` 的前置宿主，不能直接降成无关未来稿。
- `06` 的内部文档能力与发布网关处于完全不同成熟度，继续混写会误导优先级。
- `07` 的真实代码主线是数据建模与 runtime CRUD，而不是“记忆模型”。
- `08` 已经形成稳定插件信任与消费边界，不能简单丢成未来想法，但也不能写成已完成实现。

## 决策背后动机

- `03` 固定改写为 `Flow` 前置容器模块，只保留：
  - `Application` 作为一等交付单元
  - 工作台的应用列表 / 创建 / 选择
  - 应用根路由
  - 应用概览页
  - 应用内最小导航骨架
  - 单一 CTA `进入编排`
- `03` 不再把协作者、日志、监控、发布状态、复制删除等内容作为 `04` 的前置条件。
- `06` 固定拆成两个专题：
  - `06A 内部 API 文档`
  - `06B 发布网关`
- `06A` 继续以后续设计稿 `2026-04-14-settings-api-docs-on-demand-design.md` 为主真值。
- `07` 固定改名为 `数据建模、作用域与 Runtime CRUD`；“记忆”不再作为当前模块标题或实现口径。
- `08` 固定按“规则已确认，能力未完备”维护，同时保留插件来源分级、消费边界和启用审批等历史结论，不把这些稳定规则丢掉。
