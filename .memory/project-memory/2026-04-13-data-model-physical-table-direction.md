# 数据建模定义改走物理表实时生效路线

## 时间

`2026-04-13 09`

## 谁在做什么

用户与 AI 正在讨论 1Flowse 的数据建模定义底层路线，并拿本机 `../nocobase` 作为主要参考对象。

## 为什么这样做

用户希望数据建模定义更接近 NocoBase 的使用体验：在界面上新增数据表、字段、关系后，系统直接形成真实可用的数据能力，而不是先进入“发布后才生效”的元数据阶段。

## 为什么要做

当前 1Flowse 仍保留 `model definition` 发布与版本化口径，运行时数据也还停留在内存引擎；如果不尽快统一方向，后续 runtime data、插件绑定、统一 CRUD、界面配置和数据权限都会沿两套口径拉扯。

## 截止日期

未指定。

## 决策背后动机

- 数据建模定义采用“控制面保存定义 + 实时变更 PostgreSQL 物理表 + 统一运行时资源接口”的路线。
- 不保留面向用户的“发布后生效”或“版本发布控制”；用户有权限执行修改后直接生效。
- 不支持修改物理表名和物理字段类型；数据表编码与物理表名创建后固定。
- 不支持重命名物理字段；如需调整，按“删除后重建”处理。
- 删除操作允许直接执行，但需要显式确认。
- 数据建模定义与界面定义分离：
  - 数据建模定义负责数据结构、关系、校验、权限基础。
  - 字段可携带默认显示类型与基础显示配置，但这只是给动态界面层的显示提示，不直接承载页面布局逻辑。
  - 动态界面代码自行消费字段显示提示，决定在表单、列表、详情等场景如何渲染。
  - 页面 / 视图层负责列表、表单、详情、布局、区块组合和字段在不同场景下的展示方式。
- 运行时对外继续走统一路由，内部注册到具体 PostgreSQL 表仓储。
- 多租户/多团队隔离第一阶段按共享库表加范围字段与统一查询范围控制推进。
- 数据建模定义作用域在创建时固定，只允许 `team` 或 `app` 二选一。
- 结构修改同步执行，不做异步发布作业；成功后提示刷新界面数据，失败则回滚并同步返回错误。
- 第一阶段字段与关系范围固定为：
  - 字符串、数字、布尔、时间、枚举、文本、JSON
  - 单向关联
  - 一对多 / 多对一
  - 多对多
  - 唯一索引
- 第一阶段运行时数据能力固定为：
  - 列表、详情、新增、修改、删除
  - 基础筛选、排序、分页
  - 按关系展开
- 插件边界继续保持：
  - `host-extension` 可参与字段类型、实时 schema 钩子、运行时资源注册
  - `runtime extension` 仅可绑定到校验、默认值、查询范围、动作槽位
  - `capability plugin` 仅负责执行能力
- 预留外部分析 / SQL 接入能力，但第一阶段不展开完整产品能力；物理表命名、索引和系统字段先按可分析方向设计。
- “发布作业”“异步发布”不作为当前主线；若未来遇到长事务、锁表或大表迁移，再补后台作业与迁移计划能力。
